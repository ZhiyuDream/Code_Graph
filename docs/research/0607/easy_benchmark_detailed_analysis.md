## 四、核心洞察

### 4.1 Symbol Fast Path 为什么引用做得好？

今天的 Symbol Fast Path 结果（41/50 全引用）远超其他版本（最好的才 24/50），关键差异可能在于：

1. **检索结果更精准**：Symbol Fast Path 直接定位到问题相关的符号，减少了无关文件干扰
2. **上下文更聚焦**：检索到的函数数量更可控，LLM 不需要在大量文件中筛选
3. **答案生成更可控**：聚焦的上下文让 LLM 更容易分析所有相关文件

### 4.2 剩余 6 题（排除 .hpp 后）的根因

| 根因 | 题目数 | 具体 |
|------|--------|------|
| 搜到了 `common.cpp` 但没引用 | 2 题 | audit_012, audit_014 |
| 搜到了 `arg.cpp` 但没引用 | 2 题 | audit_016, audit_037 |
| 搜到了 `speculative.cpp` 但没引用 | 1 题 | audit_011 |
| `arg.cpp` 没搜到 | 1 题 | audit_020 |

**高频遗漏 .cpp 文件**：
- `common/common.cpp` — 2 题搜到未引（LLM 可能认为是"通用配置"不重要）
- `common/arg.cpp` — 2 题搜到未引 + 1 题没搜到
- `common/speculative.cpp` — 1 题搜到未引

### 4.3 没有幻觉引用

所有 50 题中，**0 题存在"没搜到但答案里引用了"的幻觉引用**。

## 五、与其他版本的对比

### 5.1 为什么 Symbol Fast Path 更好？

| 维度 | Symbol Fast Path (今天) | 最佳旧版 (06-03) |
|------|------------------------|------------------|
| 检索全召回 | 46/50 | 45/50 |
| 引用全 | **41/50** | 24/50 |
| 搜到但没引全 | **5/50** | 20/50 |

**核心差距**：旧版本有 20 题"搜到了但没引全"，Symbol Fast Path 把这个数字压到了 5 题。

**原因推测**：
- Symbol Fast Path 检索到的文件数量更少、更精准（可能 3-5 个文件）
- 旧版本 ReAct 可能检索到 10+ 个文件，LLM 上下文过载，只分析了"主要"文件
- 精准检索 -> LLM 更容易全面分析 -> 引用更完整

## 六、下一步优化方向

### 6.1 针对剩余 6 题（高优先级）

**audit_012/014**（`common.cpp` 搜到未引）：
- 根因：LLM 认为 `common.cpp` 是通用配置，与具体审计问题关联度不高
- 方案：在 answer_generation prompt 中强调"不要忽略通用配置文件，它们可能包含默认值和初始化逻辑"

**audit_016/037**（`arg.cpp` 搜到未引或没搜到）：
- 根因：`arg.cpp` 是命令行参数解析，LLM 可能认为与审计问题关系不大
- 方案：增强 grep 对 `arg.cpp` 的关键词匹配，或在 prompt 中强调参数解析文件的重要性

**audit_020**（`arg.cpp` 没搜到）：
- 根因：检索阶段遗漏
- 方案：检查 Symbol Fast Path 是否没有正确提取 `arg.cpp` 相关符号

**audit_011**（`speculative.cpp` 搜到未引）：
- 根因：LLM 可能认为 speculative sampling 是次要逻辑
- 方案：prompt 中强调"所有检索到的文件都必须分析"

### 6.2 保持 Symbol Fast Path 的优势

当前 Symbol Fast Path 的表现是历史最佳（41/50 全引用），应该：
1. **固化这个 pipeline**：把 Symbol Fast Path + 当前 answer_generation prompt 作为 baseline
2. **在此基础上微调**：只针对剩余 6 题做优化，不要大改
3. **验证稳定性**：多跑几次确认 41/50 是稳定的还是偶然波动
