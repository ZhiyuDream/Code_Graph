# 324 题 Benchmark 假阳性分析：Baseline 真的对吗？

> 分析日期: 2026-05-11  
> 核心发现: **Baseline 83.3% 和 Smart 82.4% 均被 Judge 的"找不到宽容偏差"严重高估。严格标准下两者真实能力几乎相同（72.5% vs 71.6%）。**

---

## 1. 用户质疑

> "Baseline 说'找不到'判 CORRECT，Smart 说'找不到'判 INCORRECT。那 Baseline 其实也没对？"

这个质疑直指评估标准的一致性问题。如果 Judge 对同一内容用不同尺子，那表面正确率就没有可比性。

---

## 2. 核心数据

### 2.1 Baseline 的"假阳性"

在 Baseline 被判 CORRECT 的 270 题中，有 **51 道** LLM 的回答包含"找不到/不存在/未检索到"等表述。

| 类别 | 数量 | 说明 |
|------|------|------|
| **真·假阳性** | **35 道** | 参考答案明确给出了位置/定义，但 Baseline 说找不到，Judge 却判 CORRECT |
| 合理正确 | 16 道 | 参考答案本身也说"信息不足/找不到"，Baseline 诚实承认是对的 |

**Baseline 严格正确率 = (270 - 35) / 324 = 235/324 = 72.5%**

> 表面 83.3% → 严格 72.5%，**被高估 10.8 个百分点**。

### 2.2 Smart 的"假阳性"

Smart 策略同样存在 **35 道**"找不到"假阳性。

**Smart 严格正确率 = (267 - 35) / 324 = 232/324 = 71.6%**

> 表面 82.4% → 严格 71.6%，**被高估 10.8 个百分点**。

### 2.3 严格标准下的对比

| 指标 | Baseline | Smart | 差异 |
|------|----------|-------|------|
| Judge 正确率 | 83.3% (270/324) | 82.4% (267/324) | -0.9% |
| **严格正确率** | **72.5% (235/324)** | **71.6% (232/324)** | **-0.9%** |
| "找不到"假阳性 | 35 | 35 | 0 |

**结论：两种策略的真实能力几乎完全相同。Judge 的宽松判定是系统性的评分膨胀，不是偏向某一策略。**

---

## 3. Smart 的"纠错"与"退化"对称性

这是最关键的发现：

| 方向 | 数量 | 说明 |
|------|------|------|
| **Smart 纠正 Baseline 假阳性** | **31 道** | Baseline 说找不到（被判对），Smart 给出了正确答案 |
| **Smart 新增假阳性** | **31 道** | Baseline 正常回答或答错，Smart 说找不到（被判对） |

**净效果 = 0。Smart 没有带来任何净提升。**

### 3.1 Smart 新增假阳性的构成

在 31 道 Smart 新增假阳性中：

- **24 道**：Baseline 是真正答对了的（非"找不到"），Smart 看了更多文件反而说"找不到"
  - 这是**信息干扰/注意力分散**的直接证据
  - 更多上下文反而让 LLM 失去了对核心信息的聚焦
- **7 道**：Baseline 是 INCORRECT，Smart 说"找不到"但 Judge 宽容判对

---

## 4. 典型案例

### 案例 1：Smart 成功纠正（Baseline 假阳性 → Smart 真对）

**qa_77**: "什么是 MMQ_X_Q4_0_RDNA1？"

- **参考答案**: "定义为 64，用于 Q4_0+RDNA1 路径的 X 向参数..."
- **Baseline**: "未直接包含 MMQ_X_Q4_0_RDNA1 的定义" → Judge: **CORRECT** ❌（假阳性）
- **Smart**: 详细分析了该宏的定义、功能和使用场景 → Judge: **CORRECT** ✅

→ Smart 通过查看完整文件成功找到了 RAG 片段没覆盖到的定义。

### 案例 2：Smart 退化（Baseline 真对 → Smart 假阳性）

**qa_17**: "为什么在设计 llama-hparams 时选择了当前的架构？"

- **Baseline**: 详细分析了 llama-hparams 的架构设计选择 → Judge: **CORRECT** ✅
- **Smart**: 虽然也被判 CORRECT，但回答中出现了"找不到"的表述，属于假阳性

→ Smart 看了更多文件后，注意力被分散，回答质量没有提升。

### 案例 3：Judge 双向不一致

**qa_80**: "WEBGPU_SET_ROWS_ERROR_BUF_SIZE_BYTES 的依赖关系"

- **参考答案**: "该常量将 set_rows 错误缓冲区固定为4字节..."
- **Baseline**: "未找到该常量" → Judge: **INCORRECT**
- **Smart**: "未发现该宏的定义" → Judge: **CORRECT**

→ 同一类"找不到"答案，Baseline 被判错，Smart 被判对。Judge 标准完全不一致。

---

## 5. 深层原因分析

### 5.1 Judge 判定逻辑的问题

Judge prompt 中隐含了**策略感知偏差**：

- **Baseline（函数片段）**: Judge 预期"信息有限"，所以"找不到"= 合理推断 → CORRECT
- **Smart（完整文件）**: Judge 预期"信息完整"，所以"找不到"= 未能准确回答 → INCORRECT

但这种预期是不稳定的：
- 有时候 Judge 对 Smart 也很宽容（如 qa_80）
- 有时候 Judge 对 Baseline 很严格

**Judge 的判定标准不是基于"答案内容 vs 参考答案"的客观匹配，而是基于"答案内容 vs 对策略的预期"的主观判断。**

### 5.2 为什么所有优化尝试都失败

| 优化方向 | 效果 | 根本原因 |
|----------|------|----------|
| Smart Full-files | 82.4% → 71.6% | 31 纠错 vs 31 退化，净效果为零 |
| 分层策略 | 79.9% | 阶段3重新生成推翻初稿 |
| 只提取证据 | 79.9% | Judge 阈值差异 |

**根本原因**：
1. **324 题的问题类型**主要是开放式定位/查找，函数片段通常已足够
2. **Judge 的宽松判定**让"找不到"类答案大量被判对，掩盖了真实能力
3. **信息干扰** > 信息帮助：更多上下文引入噪音，LLM 注意力分散

---

## 6. 结论与建议

### 6.1 对 Baseline 83.3% 的重新评估

**Baseline 83.3% 是一个被高估的指标。**

- 真实严格正确率约 **72.5%**
- 约 **35/270 = 13%** 的"正确"答案是"找不到"假阳性
- 这些假阳性不是 LLM 能力的体现，而是 Judge 宽容的产物

### 6.2 对 Smart 策略的重新评估

**Smart 没有失败，但也没有成功。**

- 它确实纠正了 31 道 Baseline 的假阳性（证明完整文件有信息价值）
- 但它也退化了 24 道 Baseline 原本做对的题（证明信息干扰真实存在）
- 加上 Judge 不一致，净效果为零

### 6.3 下一步建议

1. **修复 Judge 判定标准**
   - 对"找不到"类答案采用统一严格标准
   - 如果参考答案有明确位置，"找不到"= INCORRECT（无论策略）
   - 如果参考答案也说"找不到"，"找不到"= CORRECT

2. **重新评估所有实验**
   - 用严格 Judge 重新跑 Baseline 和 Smart
   - 真实能力差距可能在 ±1-2% 以内，任何"优化"都可能是噪音

3. **聚焦"真正找到"的能力**
   - 不再优化"如何优雅地说找不到"
   - 优化检索精度：让 RAG 召回真正包含答案的片段
   - 优化 LLM 的提取能力：从片段中准确提取定义和位置

4. **v2 代码审查 benchmark 才是 full-files 的适用场景**
   - 324 题的开放式定位问题不需要跨文件理解
   - v2 的调用方契约、资源生命周期分析才需要
