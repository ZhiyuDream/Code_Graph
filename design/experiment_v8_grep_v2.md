# 实验: Grep V2 (Claude Code 风格) 优化

## 日期
2026-04-22

## 改进内容
将 `grep_codebase` 从 V1（正则解析文本）替换为 V2（Claude Code 风格）：

| 特性 | V1 | V2 (Claude Code 风格) |
|---|---|---|
| 解析方式 | 正则解析 `file:line:content` | `rg --json` 结构化解析 |
| `--max-columns` | ❌ 未设置 | ✅ 默认 500 截断 |
| 输出模式 | 仅 content | content / files_with_matches / count |
| 排序 | 无 | ✅ mtime 降序（最近修改优先） |
| `-Fw` 标识符 | ❌ 仅 `-i` | ✅ `-Fw` 全词+固定字符串 |
| 排除规则 | 仅 `-tcpp` | ✅ VCS/build/CMakeFiles/node_modules |
| 分页 | Python 截断 | ✅ head_limit + offset |

## 假设
1. `--json` 解析更稳定，避免特殊字符（如 `:` 出现在代码中）导致解析错误
2. mtime 排序让最近修改的代码优先，可能提升与当前开发相关问题的准确率
3. `--max-columns` 截断减少 minified/长行内容污染上下文

## 实验配置
- 数据集: 360题 (results/qav2_test.csv)
- 并行: 20 workers
- 文件扩展: 启用 (FILE_EXPANSION_MAX=50)
- 对照组: V8 Baseline (V1 grep)

## Good Cases (GrepV2 做对、Baseline 做错)

**[2] llama-grammar 的内部结构**
- V2 召回更全面，提供了模块内部结构的完整上下文

**[6] falcon 依赖关系**
- mtime 排序让最近修改的相关文件优先，可能捕获了更准确的依赖信息

**[52] ggml-alloc.c 设计决策**
- `--json` 解析稳定性确保没有因特殊字符丢失匹配

**共性**: 需要**广泛召回**多个相关文件的问题，V2 的排除规则和解析稳定性帮助召回更完整

## Bad Cases (Baseline 做对、GrepV2 做错)

**[4] hunyuan-dense 角色**
- Baseline 恰好召回了关键文件，V2 的 mtime 排序改变了顺序，关键文件被 limit=3 截断

**[10] unary-ops 设计权衡**
- 需要精确的单文件分析，V2 的 mtime 排序引入了不同文件顺序

**[54] quants.c 关键变量位置**
- 需要精确的定位信息，排序变化导致结果不同

**共性**: 需要**精确聚焦**特定文件的问题，mtime 排序改变了结果顺序，可能将关键匹配挤出 limit

## 关键发现

### 1. 准确率基本持平 (+0.3%)
- Grep V2 的改进没有带来显著准确率变化
- 36 vs 35 的盈亏几乎抵消
- 说明 grep 在整体 pipeline 中不是瓶颈（语义搜索 + file expansion 已召回足够证据）

### 2. mtime 排序的双刃剑
- 帮助: 最近修改的文件往往更相关（+1题）
- 伤害: 改变了结果顺序，可能挤出关键匹配（-1题）
- 净效果: 几乎为零

### 3. `--json` 解析的稳定性收益未显式体现
- 测试验证了解析稳定性（含冒号等特殊字符不崩溃）
- 但 360 题中未遇到因 V1 解析错误导致的问题
- 收益是"预防性"的，而非"修复性"的

## 结论

**采纳 Grep V2，但准确率收益有限**

理由:
1. 准确率持平（+0.3%），无负面影响
2. `--json` 解析更稳定，为后续复杂查询提供保障
3. 新增 `files_with_matches` / `count` / `grep_identifier` 等工具，增强灵活性
4. 代码质量提升（结构化解析替代正则文本解析）

**建议**: 将 `grep_search_v2.py` 设为默认，保留 V1 作为回退。mtime 排序在特定场景下有价值，但非决定性改进。
