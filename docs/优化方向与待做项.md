# QA 流水线优化方向与待做项

本文档基于当前 `qa_retrieval_stats.md` 的统计结果和流水线实现，整理可考虑的优化方向，便于按优先级推进。**所有结论均来自现有数据，不编造数字。**

---

## 一、当前数据摘要（来自 396 题统计）

| 指标 | 数值 |
|------|------|
| 总体正确率（≥0.5） | 42.9%（170/396） |
| 总体正确率（≥0.6） | 20.2%（80/396） |
| 平均评价分 | 0.42，中位数 0.40 |
| 证据召回率（平均） | 0.16；中位数 0 |
| Evidence 文件数/题（平均） | 87.91 |
| 检索涉及文件数/题（平均） | 1.97 |

**按问题类型**：how 最好（57.6% ≥0.5），what 最差（32.3%）；**按一级分类**：模块级最好（66.7%），代码级最差（22.2%）。  
**路由分布**：A 38.4%，B 39.4%，C 22.2%。

## 二、图本身质量评估（独立维度）

**动机**：若图里本来就没有 Evidence 中的文件/函数或相关 CALLS，再好的检索也拿不到这些信息。因此需要把「图是否覆盖题目所依赖的实体与关系」单独评估，才能区分：**是检索没找到，还是图里根本没有**。

**可做的评估**：

1. **Evidence 文件在图中的覆盖率**
   - 对每题（或对全量 Evidence 去重）：取 Evidence 列解析出的文件路径集合，在 Neo4j 中查询有多少个路径对应「至少一个 Function 的 file_path」（或与图中 `file_path` 能匹配上）。
   - 指标示例：**Evidence 文件覆盖率** = 图中有节点的 Evidence 文件数 / 该题 Evidence 文件数；可汇总为全量或按题型/一级分类的均值、中位数。
   - 若某类题覆盖率极低，说明图构建阶段就未包含这些文件（解析范围、过滤规则、路径归一化等），需优先改进建图而非检索。

2. **实体名称在图中的可匹配性**
   - 用与 run_qa 相同的实体归一化逻辑，对每题的「实体名称」在图中查是否存在匹配的 Function（如 `file_path CONTAINS entity` 或 `name CONTAINS entity`）。
   - 指标：**实体命中率** = 能匹配到至少一个 Function 的题目数 / 总题数；可按一级分类（项目级/模块级/文件级/代码级）分桶。
   - 若某层级实体命中率低，可能是图中该粒度节点缺失或命名/路径与 CSV 不一致，需检查建图时的范围与归一化。

3. **图规模与 Evidence 的对比（抽样）**
   - 抽样若干题，列出该题 Evidence 中的文件与关键函数/调用；再在图中查这些文件、函数、CALLS 是否存在。
   - 可形成简单报告：如「本题 Evidence 共 N 个文件，图中存在 M 个，缺失的为 …」，便于判断是建图漏了模块、还是 Evidence 写了未解析的路径。

4. **与证据召回的联合解读**
   - 当前「证据召回率」= 检索到的文件与 Evidence 的交集 / Evidence 文件数。若先算「图对 Evidence 的覆盖率」，则：
     - **图覆盖率低、证据召回低** → 瓶颈在建图，应优先扩大解析范围或补全图中节点/边。
     - **图覆盖率高、证据召回低** → 瓶颈在检索策略（A/B/C 的 Cypher、BFS、embedding 等），应优先调检索。
   - 在统计报告中可同时给出「按题型的图覆盖率」与「按题型的证据召回率」，便于对照。

**落地方式**：

- 已实现 **`Code_Graph/eval_graph_coverage.py`**（覆盖率与实体命中率）
- 已实现 **`Code_Graph/investigate_graph_scope.py`**（调查建图范围与 Evidence 差异；运行后输出 `graph_scope_investigation.md`）。

**调查结论摘要（修 Evidence 解析后）**：

- Evidence 去重后约 352 条路径（.cpp 236、.h 100、.hpp 11、.c 5）；compile_commands 316 条；图中 311 条。图与当前 compile_commands 基本一致，仅 5 个文件解析未进图。
- **Evidence 中未在图中的路径**：① **.h 共 100 条** — 图中无 .h 作为 file_path（只解析编译单元），属定义不一致；② **.cpp 62 条** — 多为 ggml-metal、ggml-sycl、ggml-virtgpu 等可选后端，当前 build 未包含；③ **.hpp 等 11 条** — 图中同样无 .hpp。结论：图本身与当前 build 范围一致；覆盖率低主要来自「Evidence 含 .h/.hpp」与「Evidence 含本次未编译的可选 .cpp」。
- 后续若要提高「图对 Evidence」的覆盖：可约定**仅按 .cpp/.c 计算覆盖率**与证据召回，或扩大 CMake 构建范围（启用所需后端）后重建图。

## 三、优化方向（按模块）

### 1. 检索质量（证据召回与覆盖面）

**现象**：证据召回率极低（平均 0.16，中位数 0），每题平均只涉及约 2 个文件，而 Evidence 平均约 88 个文件/题。

**可能原因与方向**：

- **Evidence 定义过宽**：参考答案的 Evidence 是否包含「所有相关文件」而非「最小必要集合」？若 Evidence 本身是整模块列表，召回率会天然偏低；可抽样核对 CSV 中 Evidence 与题目真实需要的文件范围。
- **类型 A 检索范围偏窄**：当前按实体名匹配 Function + CALLS，可能只返回少量节点；可考虑对「文件级/模块级」实体扩大 Cypher 范围（如按 `file_path` 前缀取整文件/整目录函数再查 CALLS）。
- **类型 B 流程展开过小**：`_WORKFLOW_DEPTH_LIMIT=5`、`_WORKFLOW_NODE_LIMIT=400`，且起点由 agent 只选 1～3 个；可尝试适度提高深度/节点上限，或增加候选起点数量、多起点并行展开再合并去重。
- **类型 C 只取 top-10 函数 + 邻域**：embedding 只取 10 个函数、CALLS 邻域再限 30 条；可尝试提高 top-k（如 20）、或对高相似度函数所在文件做「整文件内函数」扩展，以增加涉及文件数。
- **统一**：为每题从检索结果中**显式收集「涉及文件路径」**并写入结果，便于和 Evidence 对齐分析；当前已有「检索涉及文件数」，可在此基础上做「按题型的证据召回」分桶统计，定位哪类题检索最窄。

---

### 2. 路由与题型策略

**现象**：what 类正确率明显低于 how；代码级（22.2%）远低于模块级（66.7%）。

**方向**：

- **路由规则细化**：当前 `_route()` 主要按意图关键词与问题类型；可针对「what + 架构/概念」类题目单独规则或权重，优先走 B（流程/架构展开）而非 C，或对「代码级」实体加强「精确函数/变量」检索（类型 A）。
- **意图/问题类型与路由的对应表**：在 `docs/llama_cpp_QA_题目分类与检索策略.md` 中已有建议；可把「意图 → A/B/C」做成可配置表，便于按统计结果调参（例如：某意图当前多为 B，但正确率低，可尝试改为 C 或 A+B 混合）。
- **代码级题目专项**：代码级正确率 22.2%、证据召回 0；可能实体粒度较细（单文件/单函数），图中匹配不到或只命中少量节点。可单独分析代码级题目的实体名称分布，看是否需路径归一化、或 fallback 到 embedding 检索。
- **分类型 A/B 测试**：对同一批题用不同路由或不同检索参数跑两版，对比正确率与证据召回，再固化策略。

---

### 3. 流程展开（类型 B）

**现状**：起点由 LLM 从最多 50 个候选里选 1～3 个，再 BFS 沿 CALLS 展开。

**方向**：

- **起点数量与候选数**：`max_pick=3`、候选上限 50；可尝试增加候选数（若图中有更多入度为 0 的节点），或允许选更多起点（如 5 个）以覆盖多入口流程。
- **深度与宽度**：深度 5、节点 400 是否过小/过大，可依「流程类」题目抽样查看子图大小与答案质量的关系；必要时做可配置或按意图/实体动态调整。
- **多起点合并**：多起点展开后的边/节点去重已做；可考虑在合并时按「文件路径」或「模块」做简单聚类，避免检索文本过长，同时保留关键边。
- **与 Evidence 的对应**：对 B 类题单独统计「检索涉及文件」与 Evidence 重叠率，若重叠率低可优先优化 B 的展开范围或起点选择 prompt。

---

### 4. Embedding 与类型 C

**现状**：用「函数名 | file_path」做 embedding，取与问题相似度 top-10，再查 CALLS 邻域 30 条。

**方向**：

- **检索粒度**：若图中有 Function 的 summary 或 docstring，可用「summary + name + path」做 embedding，语义更丰富。
- **top-k 与邻域**：top-10、30 条边偏少时可提高；或按相似度阈值（如 >0.5）取所有超过阈值的函数再扩展邻域。
- **实体约束**：类型 C 当前是否用「实体名称」过滤候选函数？若未用，可先筛 `file_path CONTAINS entity` 或 `name CONTAINS entity` 再做 embedding，减少无关模块干扰。
- **索引**：若函数量很大，可预计算所有函数 embedding 建向量索引，检索时只做 query 向量与索引的相似度搜索，加快速度并便于调 top-k。

---

### 5. 答案生成与评估

**现状**：统一用「问题 + 检索结果（截断 4000 字）」送 LLM 生成；评估为 LLM 对「参考答案 vs 生成答案」打 0–1 分。

**方向**：

- **Prompt 分类型**：对 A/B/C 使用不同 system 或 user prompt（如 A 强调「列举调用关系即可」，B 强调「按流程顺序总结」，C 强调「结合检索到的设计/实现说明理由」），可能提升一致性与得分。
- **检索结果格式化**：当前检索文本为「函数名 + 路径 + 边」；可对 B 类增加「按调用层级缩进」或「按文件分组」，便于 LLM 理解流程与模块边界。
- **评估方式**：除 LLM 打分外，可加「证据重叠度」指标（如检索文件与 Evidence 的 Jaccard）或关键事实抽取（NLI/QA 模型）与参考答案对比，避免评估单一依赖 LLM。
- **低分样本分析**：对评价分 &lt;0.3 的题目做人工抽样，看是检索缺关键证据、还是生成偏题/啰嗦，再决定优化检索还是生成。

---

### 6. 工程与可复现

- **配置外置**：将路由规则、BFS 深度/节点上限、embedding top-k、LLM 截断长度等写入配置（如 `.env` 或 `config.json`），便于做消融与复现。
- **分阶段评估**：输出中保留「仅检索」「检索+生成」「检索+生成+评估」的中间结果，便于单独评估检索质量与生成质量。
- **按路由/题型统计证据**：在 `qa_stats.py` 中已按问题类型、一级分类统计；可增加「按路由类型」的 evidence 召回/精确率分布，观察 A/B/C 哪类检索最缺文件。

## 四、建议的推进顺序（供参考）

1. **先评估图质量**：跑图覆盖率/实体命中率，看 Evidence 与实体在图中的覆盖情况；若覆盖率低，优先改进建图或解析范围。
2. **再看证据与检索**：核对若干题目的 Evidence 与「检索涉及文件」，结合图覆盖率区分「图没有」与「检索没找到」；同时做「按路由类型」的证据召回统计。
3. **再调检索**：在不动路由的前提下，针对「图覆盖高但召回低」的题型，单点放大检索范围（A 扩大 Cypher / B 多起点或加深 / C 提高 top-k 或加实体过滤），跑一小批题看正确率与召回变化。
4. **后调路由与生成**：若某意图/题型明显更适合另一路由，再调 `_route()` 或分类型 prompt；并视情况加评估维度和低分样本分析。
