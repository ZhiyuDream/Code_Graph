# llama.cpp Issue 标签信息质量调研报告（修订版）

**调研时间**：2026-03
**数据来源**：llama.cpp GitHub 仓库（93 个标签），通过 `gh api` 获取全量标签，逐一抓取 closed issues 评估
**评估方法**：对每个标签随机抽取 5 条 closed issue（不足 5 条的全部采样），从五个维度打分：commit SHA 存在（+0.12/个，上限0.36）、文件路径引用（+0.08/个，上限0.20）、代码片段（+0.02/处，上限0.15）、body 长度（+0.04~+0.15）、expected/actual 模式（+0.10）。总分 1.0，≥0.5 为高质量（High），0.35~0.5 为中质量（Med），<0.35 为低质量（Low）。

---

## 1. 核心发现：标签不是可靠的质量过滤器

**结论：同一标签内部的信息质量差异极大，仅靠标签来决定 Issue 是否入库准确率不足。**

最直接的证据：`bug` 标签内部，最高分 0.59，最低分 0.08——差距接近 8 倍。

| Issue | 标题 | 分数 | Commit | 问题 |
|-------|------|------|--------|------|
| #21006 | Eval bug: PR20908 breaks rpc-server... | **0.59** | 17个 | 详细分析+多文件修改 |
| #21123 | Misc. bug: Docker image | **0.08** | 0 | 仅一句描述，无代码上下文 |

打标签的人（maintainers + 自动化工具 + 用户自选）的动机是"这个问题属于什么类型"，而不是"这条 issue 是否有代码问答价值"。

## 3. 标签质量分类详解

### 3.1 高可靠性：入库优先考虑

#### `regression` — 平均分 0.65，高质量率 80%

**最可靠的标签。** regression 必须 bisect 到特定 commit 才能论证，描述自然包含代码版本信息。80% 高质量，0% 低质量。

真实例子（#19779，0.71分）：
> Title: Eval bug: binbcast.cpp:200: GGML_ASSERT(s10 == 1) failed
> body=19288 chars, commits=15, snippet=True

#### 硬件后端标签 — 平均分 0.50~0.63

`AMD GPU`（0.63）、`Vulkan`（0.60）、`CUDA`（0.57）、`Nvidia GPU`（0.55）、`SYCL`（0.50）、`OpenCL`（0.47）、`Apple Metal`（0.49）、`Intel GPU`（0.40）、`RoCM`（0.58）

这些后端问题的报告者通常是深度用户，有能力提供代码上下文。但平均分差异大（0.40~0.63），不能一概而论。

#### Severity 标签 — 平均分 0.42~0.60

`high severity`（0.60）最可靠，80% 高质量；`critical severity`（0.48）次之，60% 高质量；`medium severity`（0.42）40% 高质量；`low severity`（0.35）仅 20% 高质量。

Severity 标签由 maintainers 打在真实 bug 上，可靠性高于用户自选标签。

#### `bug-unconfirmed` — 平均分 0.54

意外地高。"unconfirmed" 不是低质量信号，反而代表 maintainers 已经调查过（调查过程产生代码上下文），只是不接受或无法复现。

#### `chat parser` — 平均分 0.57

解析器问题必须有模板代码才能描述，天然包含代码上下文。60% 高质量。

#### `model` — 平均分 0.61

模型相关问题通常有 commit 或详细日志。80% 高质量。

### 3.2 中等可靠性：需要 AI 评分辅助

#### `bug` — 平均分 0.37，质量参差

本次采样 5 条中：2 高质量（0.52~0.59）、1 中质量（0.44）、2 低质量（0.08~0.20）。

关键发现：**标题前缀比标签更准**：
- `Eval bug:` 前缀 → 来自自动化评估框架，几乎必有 commit+详细日志
- `Misc. bug:` 前缀 → 质量参差，从 0.08 到 0.59 都有

#### `enhancement` — 平均分 0.20，0 高质量

5 条采样全部中低质量（0.04~0.42），没有一条达到高质量标准。增强请求多数是纯设计讨论，无代码上下文。

#### `server` / `server/api` / `server/webui` — 平均分 0.34 / 0.11 / 0.30

server 系列整体偏低：涉及 HTTP 行为的问题有代码上下文，纯配置/UX 改进则无。`server/api` 最低（0.11），5 条全部低质量。

#### `stale` — 平均分 0.27，不应自动排除

本次采样质量偏低（0.04~0.44），但 stale 只代表"14天无响应"，不代表内容无价值。之前 v1 采样曾出现 0.85 分的 stale issue。

**建议：降权而非排除。**

#### `performance` — 平均分 0.34

性能问题有量化数据时质量高（如 #18258，0.71分，body=54811），但泛泛说"很慢"的问题无代码价值。20% 高质量。

#### `question` — 平均分 0.36

有代码的问题（0.66分，17863 chars）vs 纯提问（0.08分）两极分化。需要 AI 评分区分。

## 4. 采样变异性说明

同一个标签两次采样结果可能差异很大：

| 标签 | v1采样(1~20条) | v2采样(5条/标签) | 差异 |
|------|----------------|-----------------|------|
| `regression` | 0.83 | 0.65 | -0.18 |
| `bug` | 0.76 | 0.37 | -0.39 |
| `chat parser` | 0.80 | 0.57 | -0.23 |
| `stale` | 0.52 | 0.27 | -0.25 |
| `devops` | 0.24 | 0.10 | -0.14 |

**v2 结果普遍低于 v1**，可能原因：
1. v1 采样条数更多，包含了更多"正好抽到高质量 issue"的情况
2. v2 更严格控制每标签 5 条，减少了偶然性
3. GitHub API 返回顺序与时间相关，新关闭的 issue 可能在不同位置

**结论**：单次采样不足以精确确定每个标签的"真实平均分"，但**相对排序**和**排除区间**是稳定的：
- 0.60+：可靠入库（regression、硬件后端、severity 高、model）
- 0.35~0.60：需要 AI 评分
- <0.35：排除或降权

## 5. 修订后的入库策略

### 5.1 两阶段过滤架构

```
阶段一：规则排除（快速过滤噪声）
  ✗ state = open
  ✗ label ∈ {devops, testing, refactoring, documentation, help wanted,
             duplicate, wontfix, invalid, need more info, good first issue,
             cli, script, python, server/api}
  ✗ created < 2022-01-01

阶段二：AI 内容评分（决定是否入库及权重）
  输入：Issue body + title
  特征：commit SHA、文件路径、代码片段长度、expected/actual 模式、body 字符数
  输出：quality_score ∈ [0, 1]

  ✓ 入库：quality_score ≥ 0.45
  ✗ 排除：quality_score < 0.45
```

### 5.2 标签加成（仅供参考）

| 标签 | 入库加成 | 说明 |
|------|---------|------|
| `regression` | +0.15（直接入库） | commit 几乎必需 |
| `high severity` | +0.10 | maintainer 确认过的 bug |
| `bug-unconfirmed` | +0.05 | maintainer 调查过的证据 |
| `CUDA` / `AMD GPU` / `Vulkan` / `SYCL` | +0.05 | 深度用户报告 |
| `model` | +0.05 | 模型相关代码上下文 |
| `stale` | 不扣分（时间降权） | 内容价值不受 stale 影响 |
| `enhancement` | -0.10 | 默认为 feature request |
| `server/api` | -0.15 | 本次采样全低于阈值 |

### 5.3 时间衰减（不排除，仅降权）

## 8. 已知局限

1. **采样变异性**：同一标签两次采样结果可能差 0.15~0.25 分，本报告数据为 v2 单次采样（5条/标签），精确度有限
2. **Commit 检测局限**：正则 `\b(b[0-9]{4,}|[0-9a-f]{8,40})\b` 可能漏掉非标准格式的 commit 引用
3. **样本量不足**：26 个标签只有 1~4 条样本，结论置信度有限（已标注）
4. **llama.cpp 仓库未打 tag**：commit SHA 只能说明涉及某次提交，不保证能在当前代码库中找到对应位置
5. **v2 采样整体偏低**：可能因 API 返回顺序、采样随机性、或近期 issue 质量确实偏低
