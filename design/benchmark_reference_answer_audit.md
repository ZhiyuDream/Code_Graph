# Benchmark 参考答案质量审计报告

## 审计日期
2026-04-22

## 审计方法
1. 自动化统计分析（关键词匹配、长度分布）
2. LLM辅助审查（GPT-5.4 作为审查员）
3. 人工抽查验证

## 数据集概况
- 总题数: 360
- 数据来源: `results/qav2_test.csv`

---

## 发现的问题

### 问题1: 参考答案完全为空（36题，占10%）

**严重程度: 🔴 严重**

这些题目的参考答案长度为0，即完全没有任何参考答案内容。

**典型题目:**
| 题号 | 问题 |
|-----|------|
| [36] | 在 ggml-alloc.c 中包含了哪些主要的代码结构，它们之间如何相互关联？ |
| [37] | ggml-quants.c 中包含了哪些模块或子组件？ |
| [40] | ggml.c 中包含了哪些类型的代码元素？ |
| [41] | ggml-quants.c 中包含了哪些关键的代码元素？ |
| [42] | ggml-quants.c 中的代码结构之间存在哪些依赖关系？ |
| [43] | ggml.c 中各个代码元素之间存在哪些依赖关系？ |
| [52] | 为什么在 ggml-alloc.c 中选择了当前的代码实现方式？ |
| [54] | 在 quants.c 中，关键变量或数据是在哪里被定义、修改和使用的？ |
| [62] | 在 quants.c 中，某个特定标识符是在哪里被定义的？ |
| [64] | 在 ggml-alloc.c 中，整体代码是如何组织以实现主要功能的？ |

**影响:**
- 这些题目无论模型回答质量如何，都会被判为"错误"（因为没有参考答案可对比）
- 严重拉低整体准确率
- 对模型不公平

**建议:**
- 紧急补充这36题的参考答案
- 或暂时从benchmark中移除这些题目

---

### 问题2: 参考答案过于保守（16题，占4.4%）

**严重程度: 🟡 中等**

问题要求分析模块结构、依赖关系或设计决策，但参考答案说"无法判断"、"仅能确认"或"信息不足"。

**典型题目:**
| 题号 | 问题 | 参考答案核心内容 |
|-----|------|-----------------|
| [1] | apertus 包含哪些子模块，层级关系和职责划分？ | "仅能确认 apertus 暴露了 llm_build_apertus 函数...职责仍不可判定" |
| [4] | 什么是 hunyuan-dense，扮演什么角色？ | "无法确定其职责、用法与依赖关系" |
| [5] | 什么是 glm4-moe，包含哪些功能组件？ | "无法判断内部组件与其交互关系" |
| [6] | falcon 依赖了哪些模块？ | "无法推断更多外部依赖" |
| [32] | 如何在 falcon-h1 中设计和实现核心算法？ | "无法直接给出核心算法步骤" |
| [33] | 如何在 refact 中利用API实现集成？ | "无法判断依赖注入、生命周期与异常策略" |

**分析:**

这些参考答案的保守性可能有以下原因：
1. **数据构建问题**: 构建benchmark时，某些代码片段确实信息不足
2. **刻意设计**: 可能是为了测试模型是否会"过度推断"
3. **参考答案不完整**: 构建者没有深入分析代码，直接写了"无法判断"

**但问题在于:**
- 这些问题本身要求分析，但参考答案拒绝分析
- 导致模型即使给出了合理的推断，也会被判为"错误"
- 这种"陷阱题"设计对模型不公平

**LLM审查结果:**
> "答案基本只是在强调'信息不足'，没有真正回答'包含哪些子模块/子组件、层级关系、职责划分'这三个核心点...整体质量偏低。" — GPT-5.4 审查

---

### 问题3: 参考答案与问题类型不匹配

**严重程度: 🟡 中等**

部分参考答案的粒度与问题不匹配：
- 问题问"为什么"（设计决策），答案只列出函数清单
- 问题问"如何组织"（架构），答案只说"无法判断"
- 问题问"依赖关系"，答案只说"未见依赖"

---

### 问题4: 参考答案本身存在推断

**严重程度: 🟢 轻微**

部分参考答案自己也包含推断：
- "可能意在集中入口"
- "推定仅影响后续引用"
- "名称含build暗示构建语义"

这与参考答案要求模型"不要推断"形成矛盾。

## 建议

### 短期措施
1. **排除空参考答案的36题** 从当前评估中，计算"清洁准确率"
2. **重新标注保守参考答案的16题**，补充合理的推断性答案

### 中期措施
3. **建立参考答案质量标准**：
   - 不能为空
   - 必须直接回答问题的核心
   - 允许合理推断，但需标注推断依据
4. **引入多参考评估**：同一题可由多个专家标注参考答案，取交集或并集

### 长期措施
5. **考虑用 LLM 生成参考答案的辅助方法**：
   - 让 GPT-4/GPT-5 先阅读完整代码
   - 生成参考答案
   - 人工审核修正
6. **建立参考答案版本控制**：每次更新代码库时同步更新参考答案
