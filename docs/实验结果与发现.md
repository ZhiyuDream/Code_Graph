# 实验结果与发现

## Graph-Agent vs Classic-RAG 对比实验（2026-03-21）

### 实验设置
- 数据集：`llama_cpp_QA_v3.csv`，98题（Issue驱动38、代码理解30、工作流30）
- Graph-Agent：`agent_qa.py`，LLM 自主调用图查询工具，最多6步
- Classic-RAG：`classic_rag.py`，embedding 检索 top-6 chunk（Function注解+Issue/PR文本），单次生成
- LLM：同一模型（`LLM_MODEL`），评测：LLM-judge（0-1分）+ BERTScore（embedding余弦）+ Evidence Recall

### 总体指标

| BERTScore 均分 | **0.767** | 0.755 | +0.012 |
| Evidence Recall | 0.250 | **0.870** | -0.620 |
| 平均延迟 (s) | 10.79 | **7.99** | +2.80 |

### 关键发现

1. **Graph-Agent 答案质量整体更优**（LLM-judge +3.3%，BERTScore +1.2%），优势集中在"代码理解"（+8.6%）和"工作流"（+5.4%）。这类问题需要图结构信息（调用关系、文件模块）而非平面文本检索。

2. **Evidence Recall 指标偏向 RAG**：Classic-RAG embedding 直接检索文本块，天然命中 gold evidence（issue编号、文件路径）；Agent 的工具调用轨迹在"代码理解"和"工作流"类 ev_recall=0，但这两类 Evidence 字段多为文件路径，agent 答案中未以精确字符串形式出现，导致误判。ev_recall 对 agent 偏低有指标设计偏差。

3. **Agent 弱势场景**（LLM-judge 较低）：
   - Bug修复链路（0.611）：需要 commit/PR diff 信息，当前工具仅有 Issue/PR 标题+摘要
   - 改动热点（0.620）：需要历史修改频率数据，图中暂无
   - Architecture（0.675）：大文件结构理解，工具返回函数列表但缺乏模块层级信息

4. **延迟**：Agent 比 RAG 慢约 2.8s，主因是多轮工具调用（平均2.83步）

### 改进方向

- 为 Agent 增加 commit/PR diff 检索工具，改善 Bug修复链路和改动热点
- ev_recall 计算需适配 agent，将答案文本中提及的函数名/文件也纳入匹配
- Architecture 类可增加"模块概述"工具，返回文件级别摘要而非逐函数列表
- Issue驱动类 Agent 略弱于 RAG，考虑扩充 Issue/PR 的 body 检索深度（当前截断400字符）

## 按 Evidence 类型消融：有/无 PR·Issue 外挂节点（2026-03-21）

### 背景
图中节点分两类：
- **代码节点**：Function，来自代码仓本身，有调用关系边
- **外挂节点**：Issue、PR，来自 GitHub 社区信息，通过 FIXES 边关联到代码

此分析验证外挂节点对两个系统答案质量的贡献差异。

### 分组结果

| Evidence 分组 | n | Agent judge | Agent recall | RAG judge | RAG recall |
|--------------|---|-------------|--------------|-----------|------------|
| Issue only | 29 | 0.679 | 0.724 | 0.693 | **1.000** |
| Issue + PR | 9 | 0.611 | 0.333 | 0.678 | 0.444 |
| file only（纯代码节点） | 58 | **0.891** | 0.0 | 0.814 | 0.871 |
| other/empty | 2 | 0.700 | — | 0.850 | — |

### 关键发现

1. **纯代码节点（file only，n=58）是 Graph-Agent 的主战场**
   - judge **0.891 vs 0.814**（+7.7%），Agent 靠图结构推理（调用链、模块关系）显著优于 RAG
   - RAG ev_recall=0.871 很高，但答案质量反而更低——检索到了文件文本但缺乏结构理解

2. **Issue only（n=29）：RAG ev_recall 完美，Agent 答案质量接近**
   - RAG embedding 直接命中 Issue 文本，recall=1.0；Agent 靠工具搜索 recall=0.724
   - 但 judge 差距仅 0.014（0.679 vs 0.693），说明 Agent 即使没精确命中 Evidence 字符串，答案语义质量仍相当

3. **Issue+PR 同时存在（n=9）：两系统共同短板**
   - judge 均最低（0.611 / 0.678），原因是 PR 的 diff/commit 内容在图中缺失
   - RAG recall=0.444 也偏低，说明 PR 文本本身就不足以支撑回答

### 改进方向（针对外挂节点）

**短期（不改图结构）：**
- 扩大 Issue/PR body 截断上限（当前400字符），对 Issue only 类问题可显著提升 Agent recall
- Agent 工具 `search_issues` 增加返回关联 PR 编号，避免漏查

**中期（丰富图节点）：**
- 为 PR 节点添加 `changed_files` 属性（修改了哪些文件），让 Agent 能从 PR 反查代码节点
- 为 Issue 节点添加 `labels`（bug/enhancement/performance），支持分类检索

**长期（本周核心 idea）：**
- 在建图时给 Function 节点的**代码本身加注释**（即注解），验证注解质量对 file only 类问题的提升效果
- 消融实验设计见下节

## 注解消融实验结果（2026-03-21）

### 实验设计（已修订）
原计划"无注解=置空 annotation_json"被更正为更干净的消融：
- **有注解**：工具返回原始代码片段 + annotation_json（summary + workflow_role）
- **无注解**：工具仅返回原始代码片段

控制变量：图结构、工具数量、LLM 完全一致，唯一变量是注解。
数据集：`llama_cpp_QA_v4.csv`（90题，剔除 Feature需求8题，三类各30题）
脚本：`agent_qa_ablation.py`，`--no-ann` 开关切换模式，同时记录 token_usage。

### 总体指标

| 指标 | 有注解（with_ann） | 无注解（no_ann） | Delta |
|------|-----------------|----------------|-------|
| LLM-judge 均分 | 0.798 | **0.799** | -0.001 |
| BERTScore 均分 | **0.759** | 0.756 | +0.003 |
| Evidence Recall | **0.864** | 0.847 | +0.017 |
| 平均延迟 (s) | **11.69** | 12.29 | -0.60 |
| Token 均值/题 | **3,885** | 4,052 | -167 |
| Token 总计（90题） | **349,635** | 364,704 | -15,069 |
| 平均工具步数 | 2.53 | 2.48 | +0.05 |

### 按类别 LLM-judge

| 类别 | 有注解 | 无注解 | Delta |
|------|--------|--------|-------|
| Issue驱动 (n=30) | 0.653 | **0.687** | -0.034 |
| 代码理解 (n=30) | **0.920** | 0.880 | **+0.040** |
| 工作流 (n=30) | 0.820 | **0.830** | -0.010 |

### 关键发现

**1. 整体差距极小（±0.001）**
LLM 直接读代码原文的能力足够强，并不完全依赖预生成注解。

**2. 代码理解类：注解有帮助（+4.0%）**
- Purpose：有注解 **1.000** vs 无注解 0.944（+5.6%）
- Architecture：有注解 0.738 vs 无注解 0.650（+8.8%）
- Data/Control Flow：有注解 1.000 vs 无注解 0.957
- 这类问题需要理解函数语义，注解提供了预消化摘要，减少 LLM 读代码时的歧义

**3. Issue驱动类：无注解反而略好（-3.4%）**
- Bug修复链路：有注解 0.567 vs 无注解 **0.656**（差距最大）
- 原因见 Bad Case 分析（下）

**4. Token 消耗：有注解更省（-4.1%）**
注解摘要比原始代码精炼，LLM 更快获取足够信息，减少了迭代次数。

**5. Evidence Recall：有注解略高（+1.7%）**
注解帮助 agent 更精准定位相关文件路径。

### Good Case：注解起决定性作用

**问题**：函数 `operator+` 在 llama.cpp 中扮演什么角色？为什么有这么多地方调用它？

**Evidence**：`common/peg-parser.cpp`

**参考答案**：operator+ 定义在 common/peg-parser.cpp，被 26 个函数调用，是系统基础设施。Concatenate a C-string literal with a common_peg_parser object to produce a new parser.

---

**有注解回答（judge=1.0，BERTScore=0.7225）**：

> 函数 operator+ 在 llama.cpp 的 common/peg-parser.cpp 中主要用于构造解析器（parser）对象，具体表现为连接（拼接）一个字符串（字符指针或 std::string）与一个 common_peg_parser 对象，产生一个新的 common_peg_parser。
>
> 这个 operator+ 被重载为两个版本：
> 1. operator+(const char* str, const common_peg_parser& p)：通过调用 p 对象的 builder().literal(str) 方法构造一个解析器，再与 p 连接。
> 2. operator+(const std::string& str, const common_peg_parser& p)：直接将 std::string 转成 const char*，委托给第一个 operator+。
>
> 之所以有很多地方调用它，是因为在解析器构造过程中，字符串常量与解析器对象的连接拼接非常常见，用于形成复杂的解析表达式。

**无注解回答（judge=0.5，BERTScore=0.6314）**：

> llama.cpp 代码库中搜索到的 operator+ 函数主要位于 common/peg-parser.cpp 文件，且为 custom namespace（common_peg_parser）中的重载版本。但无法直接获取具体实现源码，可能是模板或内联代码未被完整检索。
>
> operator+ 函数一般在 C++ 中扮演的是运算符重载的角色，常用于自定义类型的加法操作……由于 llama.cpp 是一个机器学习语言模型推理项目，operator+ 多用于张量、数组或者解析规则的拼接也是合理推断。

**工具调用轨迹对比**：

| 步骤 | 有注解 | 无注解 |
|------|--------|--------|
| Step 1 | `search_functions("operator+")` → 找到 common/peg-parser.cpp, fan_in=26 | `search_functions("operator+")` → 同上 |
| Step 2 | `get_function_detail("operator+")` → 返回3行源码 **+ 注解："Concatenate a C-string literal with a common_peg_parser object"** | `get_function_detail("operator+ (common/peg-parser.cpp)")` → **未找到函数**（函数名带了文件路径，匹配失败） |
| 结果 | 注解直接给出语义，LLM 精准回答 | 代码没读到，开始泛泛猜测 |

**真正原因的两层**：
1. **表层（工具调用 bug）**：无注解版 LLM 传入了带文件路径的函数名 `"operator+ (common/peg-parser.cpp)"`，但图查询是精确匹配 `f.name`，导致找不到函数，源码根本没读到
2. **深层（注解的价值）**：即使能读到代码，`operator+` 只有3行，函数体是 `return p.builder().literal(str) + p`，没有上下文的话 LLM 仍难以理解 `common_peg_parser` 的设计意图；而注解直接给出了人类可读的功能描述，绕过了上下文理解的障碍

**规律**：注解的价值不仅是"额外信息"，更是**对短函数语义上下文的补充**——源码只有实现，注解告诉你"为什么这样实现、在系统里扮演什么角色"。

### Bad Case：注解干扰 Issue 检索

**问题**：llama.cpp 中出现了这个问题：Qwen3-Coder-Next generates prematurely EOS instead of tool call。这是什么原因？社区是怎么解决的？

**Evidence**：Issue #19513, PR #19773

**参考答案摘要**：问题源于多轮工具调用时 EOS token 被误判为生成结束信号；社区通过 PR #19773 修复，服务器端将连续响应输入项合并为单一助手消息，避免过早终止。

---

**有注解回答（judge=0.3，BERTScore=0.7117）**：

> 在 llama.cpp 社区的公开 Issue 中，**未检索到**关于 "Qwen3-Coder-Next 过早生成 EOS" 的相关问题记录。
>
> 建议您可以尝试以下方式来排查和解决该问题：
> 1. 检查模型推理时的生成配置参数，如最大生成长度、停止符号配置……
> 2. 在 GitHub 上搜索类似模型名称或关键词……
> 3. 提出新的 Issue 详细描述问题，寻求社区帮助。

**无注解回答（judge=0.7，BERTScore=0.7135）**：

> 在 llama.cpp 中，使用 Qwen3-Coder-Next 模型时，出现了生成过早的 EOS，导致没有正确调用工具。
>
> 出现此问题的环境：llama.cpp 版本 7993（commit 6d0ee8d4d）、Vulkan 和 CPU 后端、AMD Strix Halo 128 GB……
>
> **社区解决方案**：该问题由 PR #19773 修复。该 PR 解决方案是在服务器端合并连续的响应输入项为一个单一的助手消息，从而避免了过早产生 EOS 标志，确保工具调用可以正常进行。建议升级到包含 PR #19773 的版本。

**工具调用轨迹对比**：

| 步骤 | 有注解 | 无注解 |
|------|--------|--------|
| Step 1 | `search_issues("Qwen3-Coder-Next EOS prematurely", limit=3)` → **未找到** | `search_issues("Qwen3-Coder-Next generates prematurely EOS", limit=5)` → **Issue #19513 命中** |
| Step 2 | `search_issues("EOS prematurely")` → **未找到** | `get_issue_detail("19513")` → 获取完整问题和 PR #19773 |
| Step 3 | `search_issues("tool call prematurely")` → **未找到**，放弃，开始编建议 | — |

**真正原因**：

`search_issues` 使用 Cypher `CONTAINS` 子串精确匹配，有注解版第1步搜的关键词是 "Qwen3-Coder-Next EOS prematurely"，而 Issue 实际 title 是 "Eval bug: Qwen3-Coder-Next generates **prematurely EOS** instead of tool call"——词序不同，连续子串匹配失败。无注解版搜了更长的完整句子但刚好命中（词序与 title 一致）。

这本质是 **LLM 生成关键词策略的随机性**导致的，与有无注解无直接关系。有注解和无注解版本在 Issue 检索这一步用了不同关键词，是偶然差异，不是注解造成的系统性影响。

**改进方向**：`search_issues` 改为分词 OR 匹配（多个关键词任一命中即可），消除关键词长短对命中率的影响。

### 更深层的结论：注解是 Token 效率的基础设施

从 idx=34（Architecture 类）的轨迹可以看到一个更本质的现象：

**有注解版**：`get_file_functions` 一次调用返回15个函数，每个函数附带注解摘要（1-2句话），LLM 立刻获得整个文件的语义全貌，judge=0.7

**无注解版**：`get_file_functions` 返回15个函数名+行号（无语义），LLM 被迫逐个调用 `get_function_detail` 读源码，花了6步——其中 `llama_sampler_sample`（68行）和 `llama_sampler_backend_support`（64行）超过 `CODE_MAX=1500字符` 被截断，LLM 没读到函数体核心逻辑，judge=0.3

**结论**：注解的价值不仅是"让 LLM 更好理解代码"，更本质的是**在有限 context window 内实现大型代码库的高效覆盖**：
- 一条注解（~50 tokens）≈ 50-100行源码（~500-1500 tokens），信息密度提升10-30倍
- 超长函数（>50行）在无注解模式下必然被截断，核心逻辑丢失；注解完全绕过这个问题
- 对于需要扫描多个函数的问题（Architecture、Purpose），注解是唯一可行的方案

这说明：**注解不是锦上添花，而是让 Agent 在大型代码库中有效工作的必要条件**，尤其是对于超长函数和需要同时理解多个函数的问题。

### 结论与下一步

**结论**：
1. 注解对代码理解类问题有显著帮助（+4%），尤其是语义歧义函数和超长函数
2. 注解的核心价值是 **token 效率**：把 50-100 行代码压缩为 1-2 句摘要，使 Agent 在 context window 内高效覆盖大型代码库
3. 有注解更省 token（-4.1%），印证了信息密度提升的判断
4. Bad Case（Issue 检索失败）与注解无关，是 `search_issues` 关键词子串匹配的局限

**下一步待验证**：
- 改进 `search_issues` 为多关键词 OR 匹配，修复 Issue 检索不稳定问题
- 强注解实验：在注解中加入 callers/callees 摘要，验证图结构信息对注解质量的增益
- 人工抽检注解正确率：找出错误注解样本，量化注解质量下限
