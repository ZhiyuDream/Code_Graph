# 检索策略重构建议：废弃 A/B/C 硬分类

## 当前方案的问题

```python
# 当前的硬编码路由 - 问题多多
def _route(row: dict) -> str:
    if "Dependency" in intent or "调用" in question:
        return "A"  # 太武断，可能也需要展开
    if "Architecture" in intent or "模块" in question:
        return "B"  # 什么是"架构"？边界模糊
    if "Design rationale" in intent:
        return "C"  # 纯 embedding 召回率低
```

**核心问题**：
1. **规则维护困难**：每类问题都要人工定义规则
2. **无法组合**：一个问题可能需要多种检索方式
3. **忽略上下文**：相同词在不同语境下含义不同

---

## 方案 1：LLM 自适应路由（推荐）

让 LLM 根据问题特征决定检索策略：

```python
ROUTING_PROMPT = """分析问题，选择最佳检索策略。

可用策略：
1. **entity_search**: 精确查找函数/类定义（适用：特定函数在哪里、功能是什么）
2. **call_graph**: 沿调用链展开（适用：流程、执行顺序、调用关系）
3. **directory_scope**: 目录级扫描（适用：模块组成、文件组织）
4. **semantic_search**: 语义相似度（适用：概念、设计意图、无明确实体）

可组合使用，如：entity_search + call_graph

【问题】{question}
【实体】{entity}

请输出 JSON：
{{
    "strategies": ["entity_search", "call_graph"],  // 策略列表
    "priority": "call_graph",  // 主要策略
    "reason": "问题询问流程，需要先找到入口函数再展开"
}}"""
```

**优势**：
- 灵活：LLM 理解语境，不依赖关键词
- 可组合：一个问题可用多种策略
- 可解释：输出选择理由，便于调试

## 方案 2：并行检索 + 融合（更激进）

不选择，而是**所有策略并行执行，LLM 融合结果**：

```python
def retrieve_all(question: str, entity: str) -> dict:
    """并行执行所有检索策略"""
    with ThreadPoolExecutor() as executor:
        # 并行启动所有检索
        future_entity = executor.submit(search_entity, entity)
        future_callgraph = executor.submit(expand_calls, entity)
        future_directory = executor.submit(scan_directory, entity)
        future_semantic = executor.submit(semantic_search, question)
    
    return {
        "entity_results": future_entity.result(),
        "callgraph_results": future_callgraph.result(),
        "directory_results": future_directory.result(),
        "semantic_results": future_semantic.result(),
    }

# LLM 融合阶段
FUSION_PROMPT = """基于以下多源检索结果，生成综合回答。

【问题】{question}

【实体检索结果】（精确匹配）
{entity_results}

【调用链结果】（流程展开）
{callgraph_results}

【目录扫描结果】（模块级）
{directory_results}

【语义检索结果】（相似概念）
{semantic_results}

请综合以上信息，生成完整回答。如果不同来源信息冲突，优先相信精确匹配结果。"""
```

**优势**：
- 全面：不遗漏任何可能的信息
- 冗余：多个策略互相补充
- 容错：一个策略失败还有其他

**代价**：API 调用量增加 3-4 倍

## 方案 3：渐进式检索（平衡）

先轻量级，不够再加重：

```python
def progressive_retrieve(question: str, entity: str) -> list[dict]:
    """渐进式检索，直到信息足够"""
    collected_info = []
    
    # Step 1: 精确匹配（最快）
    entity_results = search_entity(entity)
    if is_sufficient(entity_results, question):
        return entity_results
    collected_info.extend(entity_results)
    
    # Step 2: 调用展开
    call_results = expand_calls(entity)
    if is_sufficient(collected_info + call_results, question):
        return collected_info + call_results
    collected_info.extend(call_results)
    
    # Step 3: 目录扫描
    dir_results = scan_directory(entity)
    if is_sufficient(collected_info + dir_results, question):
        return collected_info + dir_results
    collected_info.extend(dir_results)
    
    # Step 4: 语义检索（兜底）
    semantic_results = semantic_search(question)
    return collected_info + semantic_results


def is_sufficient(results: list, question: str) -> bool:
    """LLM 判断信息是否足够回答问题"""
    prompt = f"""基于以下检索结果，能否回答问题？只需回答 YES/NO。

【问题】{question}

【检索结果】
{format_results(results)}

信息是否足够？"""
    
    response = llm.complete(prompt)
    return "YES" in response.upper()
```

**优势**：
- 平衡：不会过度检索
- 自适应：简单问题快速返回，复杂问题深入
- 节省：大部分问题可能 Step 1 就解决

```python
# 简化版检索 - 废弃 A/B/C
def unified_retrieve(question: str, entity: str) -> str:
    # 1. 实体精确查找
    funcs = search_entity(entity)
    if funcs:
        # 2. 对每个找到的函数展开调用链
        subgraph = expand_calls(funcs[0]["id"], depth=3)
        return format_subgraph(subgraph)
    
    # 3. 实体找不到，尝试目录级
    dir_funcs = scan_directory(entity)
    if dir_funcs:
        return format_functions(dir_funcs)
    
    # 4. 兜底语义
    return semantic_search(question)
```
