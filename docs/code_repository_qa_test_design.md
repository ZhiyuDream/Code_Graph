# 代码仓问答技术测试设计文档

## 1. 项目背景及开发内容

### 1.1 背景

在产品线长期演进过程中积累了大量存量代码，如何高效理解历史代码，尤其是对新加入的开发人员，成为一项关键挑战。在数通场景中，代码主要基于 C/C++，蕴含复杂的业务逻辑和配置信息。

当前针对代码仓库的问答能力通常依赖两种方式：
1. **程序分析生成代码图谱**：通过构建代码图谱，实现基于图结构的查询与问答
2. **大模型生成的文档总结**：大模型扫描当前代码仓，并总结代码文档，实现文档的查询与问答

### 1.2 三大核心挑战

在千万行级代码规模下，现有问答技术仍面临以下挑战：

| 挑战 | 描述 | 关键问题 |
|------|------|----------|
| **代码动态流程理解** | 代码中存在隐式的执行顺序依赖（如模块初始化流程、异常处理链路），难以通过静态分析完全还原 | 流程追溯难 |
| **碎片化知识串联** | 业务相关信息分散在函数、文件、配置表等多个位置，缺乏有效关联 | 知识碎片化 |
| **元素定位精准性** | 精确定位相关元素（适配文件、参数定义源头、配置注入点）存在模糊匹配和召回不足 | 关联元素定位模糊 |

### 1.3 开发内容

本项目聚焦研究如何在大规模代码场景下，通过流程追溯、知识关联和高效定位，实现对开发者问题的高效问答。主要包括以下两项关键技术：

#### 1.3.1 代码仓知识结构化通用表达模型

定义统一的结构化表达形式，通过结构化挖掘方法，融合代码中的动态流程与碎片化知识，涵盖：
- 元模型定义
- 关系类型体系
- 通用化适配规则

#### 1.3.2 结构化代码知识检索模型与算法

构建基于结构化知识的高效检索模型，支持对目标元素（如适配文件、参数来源）的准确、快速定位，涵盖：
- 多维度索引结构
- 自然语言查询映射规则
- 高效检索算法

## 2. 项目目标

### 2.1 功能目标

| 目标项 | 指标 | 验证方式 |
|--------|------|----------|
| 元素提取准确率 | ≥ 85% | 人工抽样验证 |
| 关系提取类型 | 2-3 类（重点为时序关系和隐性逻辑关系） | 测试用例覆盖 |
| 通用化适配 | 在两个不同规模（百万、千万行级）代码仓中验证，无需修改模型框架即可适配 | 跨仓库测试 |

### 2.2 性能目标

在相同算力、计算资源和智能体框架条件下，在数通 C/C++ 代码仓：
- **问答准确率提升 ≥ 5%**

#### 3.4.2 答案质量评估

**评估维度**：

| 维度 | 权重 | 评估标准 |
|------|------|----------|
| **准确性** | 40% | 答案内容是否正确，与代码实际是否一致 |
| **完整性** | 30% | 答案是否涵盖问题的所有方面 |
| **相关性** | 20% | 答案是否与问题强相关，无无关内容 |
| **可理解性** | 10% | 答案是否清晰易懂，结构合理 |

**质量等级划分**：

| 等级 | 分数范围 | 描述 |
|------|----------|------|
| 优秀 | 0.9 - 1.0 | 答案完全正确，完整且清晰 |
| 良好 | 0.7 - 0.9 | 答案基本正确，略有遗漏或不够完整 |
| 一般 | 0.5 - 0.7 | 答案部分正确，有明显遗漏 |
| 较差 | 0.3 - 0.5 | 答案大部分错误，仅有少量相关信息 |
| 差 | 0.0 - 0.3 | 答案完全错误或无关 |

### 6.1 必须满足（P0）

- [ ] 元素提取准确率 ≥ 85%

- [ ] 时序关系提取准确率 ≥ 75%
- [ ] 隐性逻辑关系提取准确率 ≥ 70%
- [ ] 千万行级代码库成功适配
- [ ] 并发10用户响应时间 ≤ 10秒

| 术语 | 定义 |
