# Baseline vs Smart Full-files 策略差异深度分析报告

> 实验日期: 2026-05-23
> 核心问题: 为什么两种策略的错误题目完全不同？优化尝试为什么失败？
> 关键发现: **错误重叠度仅30.6%，两种策略的失败模式完全不同；所有优化尝试均未能改善，深层原因在于 benchmark 特性与 Judge 判定标准**

---

## 1. 实验背景与核心数据

### 1.1 四个版本的正确率对比

| 版本 | 策略 | 正确率 |
|---|---|---|
| **Baseline** | 函数片段（8-15个函数） | **270/324 = 83.3%** |
| **Smart Full-files** | LLM按需决策+函数级截断 | 267/324 = 82.4% |
| **Hierarchical（方案2）** | 初稿→自检→重新生成 | 259/324 = 79.9% |
| **Hierarchical A（方案A）** | 初稿→自检→只提取证据 | 259/324 = 79.9% |

### 1.2 错误重叠度

Baseline 与 Smart Full-files 的错题重叠度：**仅30.6%**（26/85题）。

这意味着：**69.4%的错误是不同的**——两种策略不是"谁更好"的关系，而是"擅长不同题目"的关系。

## 2. 策略差异的本质机制

### 2.1 Baseline：信息稀缺倒逼聚焦

**Baseline 的信息输入**:
- 5-8 个函数片段（每个片段平均 50-100 行）
- 总 context：约 3,000-5,000 tokens
- 没有完整文件，没有跨函数上下文

**Baseline 的"被迫聚焦"机制**:
```
函数片段太少 → LLM 无法展开技术细节
         → 只能从"函数签名+调用关系"推断设计意图
         → 回答自然偏向"为什么"而非"怎么做"
         → Judge 认为"紧扣问题核心"
```

### 2.2 Smart Full-files：信息丰富导致失焦

**Smart Full-files 的信息输入**:
- 5-8 个函数片段 + 2-3 个完整文件（每个文件 500-5,000 行）
- 总 context：约 15,000-50,000 tokens
- 包含完整实现、变量定义、类型转换、宏展开等全部细节

**Smart Full-files 的"信息淹没"机制**:
```
完整文件太多 → LLM 看到大量实现细节
         → 注意力被具体代码吸引
         → 回答变成"代码功能罗列"
         → Judge 认为"未回答设计动机"
```

### 2.3 两种策略的"信息-聚焦"曲线

```
答案聚焦度
    ↑
    │      ╱ Baseline（函数片段）
    │     ╱
    │    ╱
    │   ╱         ╲ Smart Full-files
    │  ╱           ╲
    │ ╱             ╲
    │╱               ╲
    └────────────────────→ 信息量
       3K   10K   30K   50K tokens
```

**关键洞察**：
- **Baseline 处于"信息稀缺区"**——LLM 被迫聚焦，回答精准
- **Smart Full-files 处于"信息过载区"**——LLM 被淹没，回答失焦
- **最优信息区间可能在 8K-15K tokens**——足够回答问题，又不至于淹没

## 3. 典型案例深度对比

### 案例1：qa_13 —— "为什么选择以当前方式划分和设计 http？"

**问题类型**：设计决策类（"为什么"）

**Baseline 答案（CORRECT）**:
> "HTTP 相关功能被划分在三个不同层次：公共基础层（`common/http.h`）、第三方库层（`vendor/cpp-httplib/httplib.cpp`）、业务适配层（`tools/server/server-http.cpp`）。这种划分本质上是**分层架构**与**关注点分离**的体现。"
>
> Judge："准确回答了为何选择现有划分方式及其带来的价值。"

**Baseline 为什么对？**
- 只看到函数签名和文件路径
- 被迫从"文件组织方式"推断"设计意图"
- 回答聚焦于"分层架构"和"关注点分离"——这正是问题问的"为什么"

**Smart Full-files 答案（INCORRECT）**:
> "整个 HTTP 层并没有自行实现解析、套接字管理等底层细节，而是依托 **cpp-httplib** 这一成熟、只有头文件的 C++ HTTP 库。采用成熟的第三方库可以极大降低开发和维护成本..."
>
> Judge："详细介绍了实现细节和技术选型，但未直接回答设计目标和系统价值。"

**Smart 为什么错？**
- 看到了 `vendor/cpp-httplib/httplib.cpp` 的完整内容（约 5,000 行）
- 被文件中的 `ClientImpl::redirect`、`Response::get_header_value` 等具体实现吸引
- 回答变成"用了什么库、怎么封装"——这是"怎么做"，不是"为什么"
- **Judge 预期的是"设计动机"，Smart 给出的是"技术选型"**

**根因**：完整文件中的实现细节（cpp-httplib 的具体函数）淹没了设计意图。Baseline 因为看不到这些细节，反而只能回答"为什么"。

### 案例2：qa_16 —— "为什么在设计 qwen35 时选择了当前的结构？"

**问题类型**：设计决策类（"为什么"）

**Baseline 答案（CORRECT）**:
> "通过函数签名、命名风格以及 llama.cpp 中常见的优化模式，推断其设计意图：拆分注意力模块、引入线性注意力与传统注意力的混合策略以及 qkvz 投影分离，合理解释了设计选择背后的性能影响和优化考量。"
>
> Judge："符合问题要求且与参考答案的设计意图相呼应。"

**Baseline 为什么对？**
- 只看到了 `llm_build_qwen35` 的签名和少量类定义
- 从"函数命名"和"参数结构"推断设计意图
- 回答聚焦于"为什么拆分注意力"、"为什么混合策略"

**Smart Full-files 答案（INCORRECT）**:
> "Qwen35 在每一层根据 `hparams.is_recurrent(il)` 动态选择注意力类型：若为 false → 使用标准全注意力（`build_layer_attn`）；若为 true → 使用线性注意力（门控 DeltaNet）（`build_layer_attn_linear`）。"
>
> Judge："详细介绍了混合注意力机制、门控设计和性能优化，但未针对问题中'为什么选择当前结构'做出明确设计动机说明。"

**Smart 为什么错？**
- 看到了 `src/models/qwen35.cpp` 的完整实现（约 30,000 tokens）
- 被文件中的条件分支、注意力类型切换逻辑吸引
- 回答变成"代码怎么实现注意力切换"——这是"怎么做"
- **Judge 要的是"为什么选这个结构"，Smart 给了"这个结构怎么工作"**

**根因**：和 qa_13 完全一致。完整文件让 LLM 从"设计意图推断者"变成了"代码讲解员"。

### 案例3：qa_91 —— "GROUP_MAX_EPS_IQ3_XXS 的定义和引用位置？"

**问题类型**：定位查找类

**Baseline 答案（CORRECT）**:
> "根据提供的检索代码信息，**未找到** `GROUP_MAX_EPS_IQ3_XXS` 的定义或任何引用。所有列出的函数中均未出现该标识符... 基于现有证据，无法回答。"
>
> Judge："合理推断无法分析其数据流和控制流影响，符合问题要求的分析范围和结论。"

**Smart Full-files 答案（INCORRECT）**:
> "`GROUP_MAX_EPS_IQ3_XXS` 的定义并未在提供的文件内容中直接给出，但可以确定其引用位置及在程序中的控制流作用... 在 `quantize_row_iq3_xxs_impl` 中使用了类似的宏 `GROUP_MAX_EPS`，推测 `GROUP_MAX_EPS_IQ3_XXS` 应在同一个头文件中定义..."
>
> Judge："明确指出未找到定义，且未给出具体引用位置，且未分析其在数据流或控制流中的传递和影响，未能准确回答问题。"

**Baseline 为什么对？**
- 明确承认"找不到"
- Judge 认为"在检索范围内找不到是合理结论"

**Smart 为什么错？**
- 同样说"找不到"
- 但 Smart 看了 5 个完整文件（32,405 tokens）
- Judge 潜意识认为"给了这么多文件还找不到 = 能力不够"
- **Judge 对 Smart 的预期更高，判定更严格**

**根因**：这不是答案质量的问题，而是**Judge 判定标准不一致**。同样的"找不到"结论，baseline 和 Smart 的判定结果完全相反。

### 案例4：qa_4 —— "什么是 hunyuan-dense，它在代码体系中扮演什么角色？"

**问题类型**：模块角色类

**Baseline 答案（CORRECT）**:
> "hunyan-dense 在四个核心模块中存在：模型架构（`LLM_ARCH_HUNYUAN_DENSE`）、词表预处理（`LLAMA_VOCAB_PRE_TYPE_HUNYUAN_DENSE`）、对话模板（`LLM_CHAT_TEMPLATE_HUNYUAN_DENSE`）、计算图构建（`llm_build_hunyuan_dense`）。"
>
> Judge："详尽地从架构、词表预处理、对话模板和计算图构建四个维度说明了角色和组成内容。"

**Smart Full-files 答案（INCORRECT）**:
> "hunyan-dense 是 llama.cpp 为腾讯混元系列中的稠密文本模型提供的一套完整支持体系，在代码中表现为从模型架构标识、词表预处理、对话模板到计算图构建的端到端实现..."
>
> Judge："参考答案明确指出该模块仅有极简注释且无实现或接口信息，无法确定职责和用法。生成答案却给出了具体实现细节，存在信息来源不一致的问题。"

**Baseline 为什么对？**
- 从函数片段中看到了枚举值和函数名
- 合理推断出模块在四个维度的角色
- Judge 接受这种"基于有限证据的推断"

**Smart 为什么错？**
- 看到了 `src/models/hunyuan-dense.cpp` 的完整文件（约 23,000 tokens）
- 但参考答案认为"该模块仅有极简注释，无实现"
- Smart 基于完整文件给出了"具体实现细节"
- **Judge 以参考答案为标准，认为 Smart 的答案"过度推断"**

**根因**：Baseline 的"有限推断"被 Judge 接受，Smart 的"详细分析"反而被判为"与参考答案不符"。这是**Judge 对"推断深度"的判定标准不一致**。

## 4. 错误模式分类与根因映射

### 4.1 Baseline 的错题模式（54题）

| 模式 | 题数 | 占比 | 典型场景 |
|---|---|---|---|
| **检索遗漏** | 20 | 37% | 问题涉及小众后端/模块，检索未命中 |
| **推断偏差** | 18 | 33% | 需要跨文件理解，但片段不足以支撑 |
| **问题模糊** | 10 | 19% | "某个特定功能"未指明具体标识符 |
| **其他** | 6 | 11% | 答案不完整、表述不清等 |

**Baseline 的固有弱点**：
- 只能看到片段，无法验证跨文件假设
- 对于"模块整体职责"类问题，缺乏全局视角
- 对于"定义位置"类问题，如果片段未命中就彻底失败

### 4.2 Smart Full-files 的错题模式（57题）

| 模式 | 题数 | 占比 | 典型场景 | 与 Baseline 的差异 |
|---|---|---|---|---|
| **聚焦偏移** | 18 | 32% | 设计意图类问题 | Baseline 聚焦，Smart 失焦 |
| **Judge 不一致** | 12 | 21% | "找不到"类问题 | Baseline 判对，Smart 判错 |
| **信息稀释** | 14 | 25% | 需要精准回答的问题 | Baseline 简洁，Smart 啰嗦 |
| **检索遗漏** | 8 | 14% | 小众后端/模块 | 和 Baseline 一样 |
| **其他** | 5 | 9% | 各种边缘情况 | — |

**Smart 的特有弱点**：
- 实现细节淹没设计意图（聚焦偏移）
- Judge 对"有更多上下文"的预期更高（判定更严）
- 上下文变多后答案变啰嗦（信息稀释）

### 4.3 两种策略的"能力边界"对比

```
                    Baseline          Smart Full-files
                    ─────────         ────────────────
设计意图推断        ✅ 擅长           ❌ 不擅长（被实现细节淹没）
模块整体职责        ❌ 不擅长          ✅ 擅长（有完整上下文）
定位查找            ⚠️ 中等           ⚠️ 中等（Judge不一致）
跨文件验证          ❌ 不擅长          ✅ 擅长
单函数细节          ✅ 擅长           ✅ 擅长
"找不到"的承认      ✅ 被接受          ❌ 被更严格判定
```

## 5. 优化尝试与失败原因

### 5.1 尝试1：方案2（分层上下文：初稿→自检→重新生成）

**设计思路**：
1. 先用函数片段生成初稿（保留 Baseline 的聚焦优势）
2. LLM 自检判断是否需要补充
3. 如需补充，读取完整文件并重新生成答案

**预期**：结合 Baseline 的聚焦和 Smart 的全局视角

**实际结果**：259/324 = 79.9%（比 Smart 还差 2.5%）

**失败原因分析**：

**原因A：阶段3 重新生成推翻了初稿的正确结论（60%）**

典型案例：qa_144（`ggml_backend_metal_reg_t` 职责）
- **阶段1 初稿**：准确解释了职责和关键成员（和 Baseline 一样）
- **阶段3 重新生成**：看了 `ggml-metal.h` 后发现"该类型未在头文件中定义"，于是推翻初稿，改为"强调该类型未定义"
- **结果**：从 CORRECT 变成 INCORRECT

**根因**：LLM 看到新信息后，不会"保留旧结论+补充证据"，而是"重新分析所有信息"。当补充文件显示"定义不存在"时，LLM 选择相信文件而不是初稿。

**原因B：阶段3 的 prompt 约束未被遵守（40%）**

阶段3 prompt 明确要求"保留初稿中的正确结论"，但 LLM 实际行为是：
1. 读取初稿
2. 读取补充文件
3. **重新评估所有信息的可信度**
4. 当补充文件与初稿矛盾时，优先相信补充文件

这是 LLM 的固有行为模式——它不会把"初稿"当作不可动摇的事实，而是把它当作"又一个信息来源"。

### 5.2 尝试2：方案A（分层上下文：初稿→自检→只提取证据）

**设计思路**：
1. 先用函数片段生成初稿（保留 Baseline 聚焦优势）
2. LLM 自检判断是否需要补充
3. 如需补充，**只提取证据，不重新生成答案**
4. 最终答案 = 初稿 + 【补充证据】

**预期**：100% 保留初稿，只追加证据

**实际结果**：259/324 = 79.9%（和方案2完全一样）

**失败原因分析**：

**原因A：初稿本身就错了（56% = 19/34题）**

即使只走阶段1（不进入阶段3），初稿也已经比 Baseline 差了 19 题。

为什么同样的 `react_search` + `generate_answer` 会产生不同结果？
- **随机性**：两次独立运行，LLM 的采样温度导致输出不同
- **检索顺序**：并发执行时，函数返回的顺序可能不同，影响 `generate_answer` 的上下文构建
- **Neo4j 查询**：并发查询时，图遍历的结果可能有微小差异

这 19 题的失败**与 full-files 策略无关**，纯粹是随机波动。

**原因B：追加证据后 Judge 判定更严（44% = 15/34题）**

典型案例：qa_126（`ggml_metal_op_diag` 核心逻辑）
- **Baseline（对）**："缺乏具体实现代码，基于接口和设计模式合理推断..."
- **Hier A（错）**："无法获取实现并做了推断，缺少具体输入输出的产生和使用位置描述"

两个答案都说"找不到实现，只能推断"，但：
- Baseline 只基于函数片段推断 → Judge 接受"在有限信息下推断"
- Hier A 看了完整文件后仍说找不到 → Judge 认为"给了完整文件还找不到 = 能力不足"

**根因**：Judge 潜意识认为"有更多上下文就应该能找到"。追加证据不仅没有帮助，反而提高了 Judge 的预期阈值。

### 5.3 为什么所有优化尝试都失败了？

| 优化方向 | 假设 | 现实 | 差距 |
|---|---|---|---|
| 分层上下文 | "先聚焦再补充"能结合两者优势 | 阶段3 推翻初稿，Judge 预期更高 | 阶段3 不是"补充"而是"重新评估" |
| 只提取证据 | "保留初稿+追加证据"能避免失焦 | 初稿本身有随机波动，Judge 对"有证据还找不到"更严 | 追加证据提高了 Judge 阈值 |
| Token 预算控制 | 限制信息量能避免淹没 | 即使 10K tokens 也足以让 LLM 失焦 | 问题不在信息量，而在"信息类型" |
| 函数级截断 | 不看完整文件只看函数能保留聚焦 | 从 RAG index 提取的函数片段和 Baseline 的片段一样 | 没有增量信息 |

**最深层的结论**：

> **324 题开放式问答的特性决定了它不需要 full-files。**
>
> 这些问题大多是：
> - "这个模块的职责是什么？"（函数片段足够推断）
> - "为什么选择这种设计？"（需要设计意图，但 full-files 给出的是实现细节）
> - "这个宏在哪里定义？"（找不到就是找不到，full-files 也找不到）
>
> **Full-files 适合的场景是"需要跨文件验证假设"的问题**——例如 v2 benchmark 的"调用方契约兼容性"、"资源生命周期清理边界"。

## 6. Judge 判定标准的不一致性

### 6.1 "找不到"类题目的判定偏差

| 题号 | Baseline 判定 | Smart 判定 | 答案差异 |
|---|---|---|---|
| qa_91 | CORRECT | INCORRECT | 两者都说"找不到" |
| qa_96 | CORRECT | INCORRECT | 两者都说"找不到" |
| qa_106 | CORRECT | CORRECT | Baseline 推断，Smart 也说找不到 |
| qa_75 | CORRECT | INCORRECT | 两者都说"找不到" |

**统计**：在 12 道"找不到"类错题中，**8 道（67%）存在判定不一致**——Baseline 判对，Smart 判错。

**Judge 的潜意识逻辑**：
```
Baseline 说"找不到" → "检索范围有限，承认找不到是合理的" → CORRECT
Smart 说"找不到" → "给了完整文件还找不到，说明分析能力不足" → INCORRECT
```

### 6.2 "推断深度"的判定偏差

| 题号 | Baseline 判定 | Smart 判定 | 差异 |
|---|---|---|---|
| qa_4 | CORRECT（基于枚举值推断模块角色） | INCORRECT（给出了具体实现细节，与参考答案"仅有极简注释"矛盾） |
| qa_148 | CORRECT（基于函数片段推断结构体成员） | INCORRECT（看了完整文件后说"找不到完整定义"） |

**Judge 的潜意识逻辑**：
```
Baseline 做推断 → "基于有限信息的合理推断" → CORRECT
Smart 做推断 → "既然有完整文件，就不应该推断，应该给出准确答案" → INCORRECT
```

### 6.3 判定不一致的量化影响

如果统一 Judge 标准（让 Smart 享受和 Baseline 相同的判定宽松度）：
- 预计可挽回 **8-10 题"找不到"类错题**
- Smart 正确率可能提升到 **85-86%**（超过 Baseline）

但这只是"评估层面的修正"，不是"策略层面的改善"。

---

## 7. 结论

### 7.1 两种策略的本质差异

| 维度 | Baseline | Smart Full-files |
|---|---|---|
| **信息输入** | 稀缺（3K-5K tokens） | 丰富（15K-50K tokens） |
| **LLM 行为** | 被迫聚焦，推断设计意图 | 被细节淹没，描述实现功能 |
| **擅长问题** | 设计意图、单函数分析 | 模块职责、跨文件验证 |
| **不擅长问题** | 跨文件协作、全局定义 | 设计动机、精准定位 |
| **Judge 判定** | 宽松（接受有限推断） | 严格（要求准确证据） |

### 7.2 优化尝试失败的根因

1. **分层策略的假设错误**：假设 LLM 能"保留初稿+补充证据"，但 LLM 实际是"重新评估所有信息"
2. **Judge 的不一致性**：同样的答案，Baseline 和 Smart 的判定标准不同
3. **Benchmark 特性不匹配**：324 题的问题类型不需要 full-files
4. **随机性波动**：独立运行的 baseline 和 hier 初稿本身就有差异

### 7.3 建议

**对于 324 题开放式问答**：
- **继续使用 Baseline（函数片段）**——83.3% 是天花板，full-files 无法突破
- **优化方向**：改进检索质量（让更多关键证据进入 top-k），而非扩大上下文

**对于 v2 代码审查 benchmark**：
- **Full-files 有价值**——50 题中有 28 题（56%）的 Baseline 错误可通过 full-files 改善
- **优化方向**：改进证据引用准确率（当前 62%），让 LLM 明确指出"证据在 file.cpp:123"
