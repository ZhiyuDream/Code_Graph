### 关键洞察

1. **Neo4j扩展工具调用最多但效果最差**
   - callers/callees 合计占比 57.5%
   - 但每次只找到 ~0.5 个新函数

2. **初始检索工具效果更好**
   - semantic/graph/grep/hybrid 平均每次找到 3-6 个函数
   - **结论**：初始检索比扩展步骤更重要

3. **Fallback机制有效**
   - grep_fallback 平均找到 5.97 个函数
   - 与 graph 并列最高
   - **结论**：当Embedding不足时，Grep Fallback是有效的补充

## 三、路由策略详解

### 3.1 各策略实际包含的工具

> ⚠️ **重要说明**：策略名称不等于实际执行的工具。以下是每个策略**真实**调用的工具：

| 策略名称 | 实际执行的工具 | Issue搜索 | 备注 |
|----------|---------------|-----------|------|
| **semantic** | `semantic` + `issue` | ✅ | 纯语义搜索 |
| **grep** | `grep` | ❌ | 纯Grep文本搜索 |
| **graph** | `graph遍历` + `semantic` | ❌ | 图遍历为主，语义搜索补充 |
| **hybrid** | `semantic` + `grep` + `issue` | ✅ | 并行执行，LLM融合结果 |

**关键发现**：
- `graph` 策略**不是纯图遍历**，它会额外调用 semantic 作为补充
- `semantic+graph` 工具对的出现是因为 `graph` 策略内部包含了 semantic
- 只有 `semantic` 和 `hybrid` 策略会搜索 GitHub Issues

### 3.2 路由策略分布

| 策略(primary+secondary) | 题目数 | 占比 | 平均找到函数 |
|------------------------|--------|------|--------------|
| grep+graph | 78 | 21.7% | 3.7 |
| semantic+grep | 77 | 21.4% | 7.1 |
| graph+grep | 65 | 18.1% | 7.4 |
| hybrid+grep | 36 | 10.0% | 5.1 |
| hybrid+graph | 29 | 8.1% | 5.2 |
| grep+semantic | 26 | 7.2% | 3.7 |
| semantic | 26 | 7.2% | 6.1 |
| semantic+graph | 9 | 2.5% | 7.7 |
| 其他 | 14 | 3.9% | - |

### 3.3 关键洞察

1. **最常用策略不是最优策略**
   - `grep+graph` 最常用 (21.7%) 但效果差 (3.7个函数)
   - `graph+grep` 效果更好 (7.4个函数)，但差别在于执行顺序和触发条件
   - **结论**：路由决策有优化空间

2. **semantic+graph 策略效果最好但最少用**
   - 平均找到 7.7 个函数（最高）
   - 但只占 2.5%，可能是因为触发条件过于严格
   - **结论**：graph 策略的语义补充很有价值

3. **hybrid策略未达预期**
   - hybrid组合占 20%+
   - 但效果 (5.1-5.2个函数) 不如 `semantic+grep` (7.1个)
   - **结论**：多策略并行的融合排序需要优化

## 四、检索步数分布

| 步数 | 题目数 | 占比 |
|------|--------|------|
| 2 步 | ~78% |
| 3 步 | ~20% |
| 4 步 | ~2% |

### 关键洞察

1. **78.3% 的题目在3步内完成**
   - Step 1: 初始检索
   - Step 2: 第一次扩展
   - Step 3: 第二次扩展或停止
   - **结论**：智能停止机制有效，避免了过度扩展

2. **21.5% 的题目走到4步**
   - 可能是因为前几次扩展都有收获
   - 或者熔断机制判断延迟

3. **几乎没有超过4步的**
   - 说明递减回报检测有效

## 五、扩展步骤效果分析

| 指标 | 数值 |
|------|------|
| 总扩展尝试 | 792 次 |
| 平均每次新增函数 | 0.47 个 |
| 零增益比例 | 80.2% |
| 扩展成功率 | 36.7% |

### 关键洞察

1. **80.2% 的扩展是徒劳的**
   - 每5次扩展只有1次能找到新函数
   - **结论**：ReAct扩展的价值有限

2. **即使成功，收益也很低**
   - 成功时平均只找到 0.47 个新函数
   - **结论**：扩展步骤的ROI很低

3. **熔断机制非常必要**
   - 如果没有熔断，80%的调用都是浪费
   - 当前熔断策略有效避免了更多无效调用

## 六、工具组合分析

### 最常见工具组合（前两步执行顺序）

| 组合 | 次数 | 占比 | 说明 |
|------|------|------|------|
| graph -> semantic | 68 | 18.9% | ⚠️ `graph`策略先执行图遍历，然后内部调用semantic |
| grep -> neo4j_callers | 66 | 18.3% | Grep定位函数，再扩展其调用者 |
| grep -> neo4j_callees | 43 | 11.9% | Grep定位函数，再扩展其被调用者 |
| semantic -> grep_fallback | 40 | 11.1% | 语义搜索不足时，fallback到grep |
| semantic -> neo4j_callers | 38 | 10.6% | 语义搜索后，扩展相关函数的调用者 |

### 工具共现分析（整道题维度）

| 工具对 | 题目数 | 占比 | 说明 |
|--------|--------|------|------|
| neo4j_callers + semantic | 126 | 35.0% | ReAct扩展阶段常见组合 |
| neo4j_callees + semantic | 101 | 28.1% | ReAct扩展阶段常见组合 |
| neo4j_callees + neo4j_callers | 98 | 27.2% | 同时扩展调用者和被调用者 |
| grep + neo4j_callers | 82 | 22.8% | Grep定位后扩展调用链 |
| graph + semantic | 68 | 18.9% | ⚠️ **主要是`graph`策略内部包含semantic** |

### 关键洞察

1. **语义搜索 + Neo4j扩展是主流路径**
   - 35% 的题目使用了 semantic + callers
   - 28% 的题目使用了 semantic + callees
   - **结论**：先Embedding检索，再图扩展是标准流程

2. **Grep经常与Neo4j配合使用**
   - 22.8% grep + callers
   - 16.1% grep + callees
   - **结论**：有明确函数名时，先用Grep定位再扩展

3. **graph + semantic 高占比的原因**
   - 18.9% 的题目显示使用了 graph + semantic
   - **但这主要是因为 `graph` 策略内部会调用 semantic 作为补充**
   - 不是两个独立工具的并行使用，而是策略的内部实现
   - **结论**：`graph` 策略的设计是"图遍历为主，语义搜索为辅"

## 八、核心洞察与建议

### 8.1 当前系统的问题

1. **扩展步骤效率低下**
   - 80% 的扩展尝试是徒劳的
   - 即使成功，也只找到 0.47 个函数
   - **建议**：减少扩展尝试，或提高扩展的准确性

2. **路由决策不够精准**
   - 最常用的策略(grep+graph)效果最差
   - 最好的策略(semantic+graph)使用最少
   - **建议**：优化路由prompt，提高策略选择准确率

3. **工具调用分布不均**
   - Neo4j扩展占57%，但效果最差
   - 初始检索工具效果好但占比低
   - **建议**：调整工具使用优先级

### 8.2 优化方向

1. **减少无效扩展**
   - 当前熔断阈值：连续2轮增益≤1
   - 建议收紧为：连续1轮增益≤1就停止
   - 预计可减少30%的无效调用

2. **优化初始检索策略**
   - 推广 semantic+graph 策略（效果最好但使用少）
   - 减少 grep+graph 策略（效果差但使用多）
   - 优化路由决策逻辑

3. **考虑移除低效工具**
   - 如果扩展步骤80%都失败，是否还需要？
   - 考虑只保留：semantic + grep_fallback
   - 简化架构可能效果更好

### 8.3 数据支撑的设计决策

基于本分析，建议的简化版架构：

```python
def simplified_search(question):
    # 1. Embedding检索（核心，占80%价值）
    results = semantic_search(question)
    
    # 2. 如果相似度低，补充Grep
    if max_score < 0.5:
        entities = extract_entities(question)
        grep_results = grep_search(entities)
        results.extend(grep_results)
    
    # 3. 最多1轮扩展（当前是2-3轮）
    if need_expansion(results):
        expand_once(results)  # 只扩展一次
    
    return results
```

**预期效果**：
- 工具调用从 3.37 次/题 降到 2 次/题
- 保持或提升正确率
- 速度提升 30%
