## 二、答案一致性（Binary Judge）

用 `gpt-4.1-mini` 对 50 题做 binary judge（判断生成答案是否正确回答了问题）：

| 指标 | 数值 |
|------|------|
| **正确率** | **46/50 = 92.0%** |
| 错误 | 4/50 = 8.0% |

### 2.1 按维度拆分

| 维度 | 正确率 |
|------|--------|
| 调用方契约兼容性 | 13/13 = **100%** |
| 配置/参数传播一致性 | 11/11 = **100%** |
| 状态一致性 | 4/4 = **100%** |
| 并发/异步安全 | 4/4 = **100%** |
| 初始化/外部 API 契约 | 2/2 = **100%** |
| 失败返回语义 | 4/4 = **100%** |
| 资源生命周期/清理边界 | 4/4 = **100%** |
| **共享 helper 复用影响** | **4/8 = 50%** ← 唯一薄弱维度 |

### 2.2 错误题目详细分析

| 题目 | 覆盖率 | 问题 | 错误根因 |
|------|--------|------|----------|
| posthoc_public_021 | 33% | 通用空白裁剪 helper | 未找到 helper 函数定义，答案偏离到内嵌逻辑分析 |
| posthoc_public_022 | 67% | 只处理前导空白的 helper | 未找到该 helper 及其调用点，未给出明确判断 |
| posthoc_public_023 | 67% | 只处理尾部空白的 helper | 同上，未找到相关定义和调用 |
| posthoc_public_026 | 67% | 过滤空白片段的 helper | 转向了 `match_hostname` 和 `detail::split` 的错误分析 |

**共同模式**：4 道错误题全部是"共享 helper 复用影响"维度，且都涉及 **chat-auto-parser-helpers** 相关的字符串/片段处理 helper。LLM 无法从自然语言描述中准确定位这些 helper 函数（如 `trim`、`strip`、`filter_empty` 等），导致答案偏离。

## 三、覆盖率 vs 正确率交叉分析

| 覆盖率区间 | 题数 | 正确数 | 正确率 |
|-----------|------|--------|--------|
| **100%** | 21 | 21 | **100%** |
| **50-99%** | 20 | 17 | **85%** |
| **1-49%** | 4 | 3 | **75%** |
| **0%** | 5 | 5 | **100%** |

### 关键发现

1. **覆盖率 100% 的题全部正确** — 检索召回完全时，LLM 能给出准确答案
2. **覆盖率 0% 的题也全部正确** — 这 5 题的答案质量与检索无关，LLM 可能基于已有知识或推理能力给出了正确答案
3. **错误集中在 33-67% 覆盖率区间** — 检索到了部分 evidence 但没找全，LLM 基于不完整信息给出了偏离的答案
4. **覆盖率不是唯一决定因素** — 50-99% 区间仍有 85% 正确率，说明即使 evidence 未全召回，LLM 也能基于部分信息给出合理答案

## 四、核心洞察

### 4.1 Hard Benchmark 的正确率 > Easy Benchmark

| 维度 | Easy Benchmark | Hard Benchmark |
|------|---------------|----------------|
| 平均检索覆盖率 | ~99% | 71% |
| Binary Judge 正确率 | ~90% | **92%** |

**反直觉发现**：Hard benchmark 虽然检索覆盖率远低于 easy（71% vs 99%），但答案正确率反而略高（92% vs ~90%）。

**原因推测**：
1. Hard benchmark 的问题更"开放"（自然语言描述），LLM 有更多推理空间
2. Easy benchmark 的 audit_question 更具体（含精确函数名/行号），LLM 对 evidence 缺失更敏感
3. Hard benchmark 的参考答案本身也更宽松（常说"当前证据只能支持局部判断"），judge 标准相对宽松

### 4.2 "共享 helper 复用影响"是唯一薄弱维度

8 题中错了 4 题，正确率仅 50%。

**根因**：
- 这些 helper 函数（trim、strip、filter 等）名称不固定、可能被内联或匿名
- 自然语言描述（"通用空白裁剪 helper"）难以映射到具体符号
- 检索器无法从"helper"、"裁剪"等模糊描述中定位到准确的函数定义

### 4.3 覆盖率与正确率的非线性关系

```
覆盖率 100% → 正确率 100%（检索全召回 = 答案必对）
覆盖率 50-99% → 正确率 85%（部分召回 = 大概率对）
覆盖率 0% → 正确率 100%（零召回 ≠ 必错，LLM 可凭推理答对）
```

**结论**：对于 hard benchmark，**检索覆盖率的中位数比最大值更重要**。即使只召回 50% 的 evidence，LLM 仍有 85% 的概率给出正确答案。

## 六、下一步优化方向

### 6.1 针对"共享 helper 复用影响"维度（高优先级）

**问题**：4/8 题错误，涉及 chat-auto-parser-helpers 的字符串处理 helper

**候选方案**：
1. **增强 QueryAnalyzer**：对"helper"、"trim"、"strip"、"裁剪"等关键词做特殊处理，优先检索 `chat-auto-parser-helpers.cpp`
2. **预置知识库**：把 `common/chat-auto-parser-helpers.cpp` 中的 helper 函数名加入同义词映射
3. **semantic_search 增强**：用"string trim helper"、"whitespace strip"等英文查询替代中文模糊描述

### 6.2 提升零召回题目的检索（中优先级）

5 题零召回虽然答案都对了，但检索召回仍应提升：
- `posthoc_public_001`（设备切换）— 涉及 SYCL 设备管理
- `posthoc_public_002`（聊天模板）— 涉及 `llama_model_chat_template`
- `posthoc_public_013`（缓存扩容）— 涉及 buffer reallocation

**候选方案**：在初始搜索阶段加入系统级 embedding（用原始问题直接做语义搜索），作为 ReAct 的打底。

### 6.3 验证稳定性

当前 92% 正确率是单次运行的结果，需要：
1. 多跑几次确认正确率波动范围
2. 对比不同版本（55.7%、49.5% 覆盖率）的正确率，验证"覆盖率 vs 正确率"的关联是否稳定
