# V7 P1 版本错误题目深度分析报告

## 一、总体情况（与V7 Minimal对比）

| 指标 | P1 | V7 Minimal | 差距 |
|------|-----|------------|------|
| 错误率 | 28.9% (104/360) | 34.4% (124/360) | **-5.5%** ✓ |
| 错误题平均得分 | 0.54 | 0.52 | +0.02 |
| 检索<=3个函数的错误 | 37题 | 48题 | **-11题** ✓ |
| 走到4步的错误 | 0题 | 120题 | **-120题** ✓ |

**关键发现**: P1的完整扩展机制（callers/callees）有效降低了错误率！

---

## 二、错误原因分析

### 原因1: 检索不足（仍是最主要原因，但比V7 Minimal好）

| 检索函数数 | P1错误题数 | V7 Minimal | 差距 |
|-----------|-----------|------------|------|
| <= 3个 | 37 (35.6%) | 48 (38.7%) | -11题 |
| <= 5个 | 68 (65.4%) | - | - |
| <= 8个 | 96 (92.3%) | - | - |

**P1的优势**:
- callers/callees 扩展能补充更多相关函数
- 即使初始检索少，扩展步骤也能找到额外函数
- V7 Minimal 跳过扩展后，检索不足的题目增多

### 原因2: 步骤分布（与V7 Minimal截然不同）

| 步骤数 | P1错误题数 | V7 Minimal | 分析 |
|--------|-----------|------------|------|
| 2步 | 87 (83.7%) | 1 (0.8%) | P1大部分错误在2步就停止 |
| 3步 | 17 (16.3%) | 3 (2.4%) | |
| 4步 | 0 (0%) | 120 (96.8%) | **关键差异** |

**关键发现**:
- P1的83.7%错误题目在2步就停止，说明**初始检索后就被判定为"足够"**
- 但实际上这些题目的答案质量不高（平均0.54分）
- 可能是**智能停止条件过于宽松**，或者**答案生成阶段的问题**

### 原因3: 问题类型分布

| 问题类型 | P1错误数 | 占比 | 平均得分 | V7 Minimal对比 |
|---------|---------|------|---------|---------------|
| 代码位置类 | 49 | 47.1% | 0.54 | 44.4% |
| 如何/怎样类 | 49 | 47.1% | 0.56 | - |
| 模块/架构类 | 28 | 26.9% | 0.56 | 13.7% |
| 为什么/设计类 | 21 | 20.2% | 0.58 | - |

**最困难的问题类型**:
1. **代码位置类**（47.1%）- 需要精确定位代码位置
2. **如何/怎样类**（47.1%）- 需要理解实现机制

## 三、P1 vs V7 Minimal 关键差异分析

### 差异1: 扩展机制的有效性

| 场景 | P1 | V7 Minimal | 结果 |
|------|-----|------------|------|
| "文件X有哪些函数" | 通过扩展找到更多函数 | 跳过扩展，检索不足 | P1更好 |
| "Y的调用链是什么" | 通过callers/callees扩展 | 跳过扩展 | P1更好 |
| 初始检索好 | 2步停止 | 走到4步空转 | V7 Minimal浪费 |

### 差异2: 智能停止的触发

**P1的问题**:
- 83.7%的错误题目在2步就停止
- 说明智能停止条件**过早触发**
- 可能判定"信息足够"但实际上不足以回答好

**V7 Minimal的问题**:
- 96.8%的错误题目走到4步
- 但扩展步骤被跳过，变成**无效循环**
- 时间浪费但没有收益

### 差异3: 错误类型分布

| 类型 | P1 | V7 Minimal | 分析 |
|------|-----|------------|------|
| 模块/架构类 | 26.9% | 13.7% | P1更多 |
| 代码位置类 | 47.1% | 44.4% | 相近 |

**为什么P1的模块类错误更多？**
- P1有LLM路由决策，可能选择了不合适的策略
- 智能路由在某些架构类问题上判断失误
- V7 Minimal的固定策略反而在某些场景更稳定

## 六、核心结论

### P1相比V7 Minimal的优势

1. **扩展机制有效**: 
   - callers/callees 扩展在检索不足时能补充函数
   - 错误率降低 5.5%
   - 检索<=3个函数的错误减少11题

2. **避免无效循环**:
   - 智能停止有效，不会走到4步空转
   - V7 Minimal 120题走到4步，P1只有0题

### P1的问题

1. **智能停止过早**:
   - 83.7%的错误在2步就停止
   - 被判定为"信息足够"但实际回答质量不高
   - 停止条件需要优化

2. **路由决策失误**:
   - 模块/架构类错误占比26.9%（比V7 Minimal高）
   - LLM路由在某些场景选择了不合适的策略

3. **检索质量≠数量**:
   - 检索到7个函数但猜错具体功能
   - 需要更好的相关性排序

### 改进建议

1. **优化停止条件**:
   - 不仅看函数数量，还要看函数相关性
   - 结合问题类型判断是否需要继续扩展

2. **改进路由决策**:
   - 架构类问题优先使用 semantic + graph
   - 代码位置类问题优先使用 grep + 文件探索

3. **保留扩展机制**:
   - 证明 callers/callees 扩展有价值
   - 不应像V7 Minimal那样完全移除
   - 应该优化触发条件
