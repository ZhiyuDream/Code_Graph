# 上下文获取策略

> 描述当前 Agent 系统的上下文获取机制、截断策略及其影响。

---

## 1. 概述

当前系统的上下文获取是**基于 Neo4j 知识图谱的多步 Agent 检索**。LLM 通过工具调用（tool-calling）访问图数据库中的代码实体（函数、文件、Issue/PR），每次工具调用返回的结果受截断限制，最终在有限的 context window 内生成答案。

---

## 2. 截断机制

系统存在两重截断硬限制：

### 2.1 源码截断（CODE_MAX）

**当前状态: 已取消截断**

```
位置: agent_qa_ablation.py::_read_source()
默认值: 999999 字符（极大值，等效无截断）
```

历史值: 1500 字符 → 2026-03-23 取消

### 2.2 工具结果截断（TOOL_RESULT_MAX）

**当前状态: 已取消截断**

```
位置: agent_qa_ablation.py::run_agent()
默认值: 999999 字符（极大值，等效无截断）
```

历史值: 1500 字符 → 2026-03-23 取消

### 2.3 其他限制

| 限制 | 值 | 说明 |
|------|---|------|
| MAX_STEPS | 6 | Agent 最多执行 6 轮工具调用 |
| max_tokens | 800 | LLM 每次生成最多 800 tokens |
| prompt_tokens | 无显式限制 | 累积在 messages 列表中，受模型 context window 约束 |
| **源码截断** | **已取消** | CODE_MAX=999999 |
| **工具结果截断** | **已取消** | TOOL_RESULT_MAX=999999 |

## 3. 工具与上下文获取

### 3.1 可用工具

| 工具 | 作用 | 返回内容 |
|------|------|---------|
| `search_functions` | 按名字模糊搜索函数 | 函数名+文件+fan_in/out |
| `get_function_detail` | 获取函数详情 | 源码片段（受CODE_MAX限制）+ 可选注解 |
| `get_file_functions` | 列出文件内所有函数 | 函数列表+行号+ 可选注解摘要 |
| `get_callers` | 获取上游调用者 | 调用该函数的函数列表 |
| `get_callees` | 获取下游被调者 | 该函数调用的函数列表 |
| `search_issues` | 搜索 Issue | Issue 标题+编号 |
| `get_issue_detail` | 获取 Issue 详情 | Issue 标题+描述+关联PR |
| `semantic_search` | 语义搜索函数 | 函数名+ 可选注解摘要 |

### 3.2 有注解 vs 无注解模式

`get_function_detail` 和 `get_file_functions` 在**有注解模式**下会额外返回 `annotation_json` 中的摘要信息：

```python
# 无注解：只返回源码
body = f"--- 源码 ---\n{code}"

# 有注解：源码 + annotation_json
body = f"--- 源码 ---\n{code}"
body += f"\n--- 注解 ---\n功能摘要: {summary}"
```

注解内容由 `annotate_functions.py` 在建图时生成，包含：
- `summary`: 函数功能的一句话描述
- `workflow_role`: 在系统工作流中的角色
- `caller_context`: 调用该函数的典型场景

## 4. Agent 执行循环

```
messages = [system_prompt, user_question]

for step in range(MAX_STEPS=6):
    resp = LLM(messages, tools=TOOLS_SCHEMA)
    if not tool_calls:
        return resp.content  # 直接生成答案
    for tc in tool_calls:
        result = execute_tool(tc)
        # 注: TOOL_RESULT_MAX=999999，等效无截断
        messages.append(tool_result)
# 超过MAX_STEPS，强制生成答案
```

## 5. 截断对效果的影响（历史验证，当前已取消）

### 5.1 截断放大问题（已验证）

**案例**: idx=34（llama-sampler.cpp Architecture问题）

- 有截断（CODE_MAX=1500）：`llama_sampler_sample`（68行）和 `llama_sampler_backend_support`（64行）被截断，核心逻辑丢失，无注解版 judge=0.3
- 无截断：LLM 能读完整源码，核心逻辑完整

### 5.2 无截断下的真实差距

全量 90 条数据，无截断条件下：

| 指标 | 无注解 | 有注解 | 差值 |
|------|--------|--------|------|
| LLM-Judge | 0.7844 | 0.7978 | +0.013 |
| Evidence Recall | 0.8693 | 0.8807 | +0.011 |
| BERTScore | 0.7549 | 0.7634 | +0.009 |
| 总token | 366,910 | 323,850 | -12% |

**结论**: 截断确实放大了问题。取消截断后，两种模式正确率差距微弱，但有注解仍节省 12% tokens。

### 5.3 截断导致的具体失败模式

1. **长函数截断**: 68行函数被截成1500字符，丢失函数尾部的关键逻辑
2. **工具结果截断**: `get_file_functions` 返回15个函数的完整列表被截断，LLM 无法看到所有函数
3. **Issue详情截断**: 长Issue描述被截断，LLM 无法获取完整上下文

## 6. 评估指标

系统使用三重指标评估答案质量：

| 指标 | 计算方式 | 说明 |
|------|---------|------|
| **LLM-Judge** | LLM 对比参考答案 vs 生成答案打分 0-1 | 正确性+完整性，由独立LLM评判 |
| **Evidence Recall** | 生成答案中包含的 gold evidence 比例 | 检索质量，evidence命中数/总数 |
| **BERTScore** | 生成答案与参考答案的 embedding 余弦相似度 | 语义相似度 |

## 7. 当前策略的问题与局限

### 7.1 上下文积累问题（当前主要问题）

- messages 列表无限增长（每轮工具调用追加完整 tool result）
- MAX_STEPS=6 后强制结束，可能答案不完整
- 没有显式的 context window 管理或摘要压缩
- prompt_tokens 持续增长直到达到模型 context window 上限

### 7.2 注解的权衡

- 注解提升了信息密度（一条注解≈50tokens vs 50-100行代码≈500-1500tokens）
- 但注解可能丢失关键细节（如 idx=44 的 GGML_ASSERT 间接调用关系被忽略）
- 简洁注解在高信息需求场景反而误导 LLM
- 无截断后源码信息足够，注解的"语义补全"价值被稀释

### 7.3 截断的历史决策

| 日期 | 决策 | 原因 | 结果 |
|------|------|------|------|
| 2026-03-21 | 引入 CODE_MAX=1500 | 防止 context overflow，控制 token 成本 | 导致长函数截断丢逻辑 |
| 2026-03-21 | 引入 TOOL_RESULT_MAX=1500 | 防止单次工具返回过长 | 导致文件函数列表被截 |
| 2026-03-23 | 取消截断（CODE_MAX=999999） | 验证对正确率的影响 | 正确率提升微弱，但 token 消耗增加 |

## 9. 未来可能的改进方向

1. **自适应截断**: 根据函数重要性（fan_in/fan_out）决定截断阈值
2. **分层摘要**: 对于超长函数，先返回摘要，LLM 需要时再请求完整源码
3. **Context 压缩**: 对 messages 列表做增量摘要，释放 context 空间
4. **截断感知**: 工具返回时标记"已截断"，让 LLM 知道信息不完整
