# Code_Graph 建图方法分析与改进建议

## 一、当前建图方法概述

### 1.1 Pipeline 架构

```
阶段1 (run_stage1.py): AST解析 → 图构建 → Neo4j写入
阶段2 (run_stage2.py): 入口发现 → 调用链展开 → Workflow写入
阶段3 (run_stage3.py): GitHub API → Issue/PR写入
```

### 1.2 两种AST解析方案

| 方案 | 实现 | 优点 | 缺点 |
|------|------|------|------|
| libclang | `ast_parser.py` | 批量解析快，无需外部进程 | 跨文件调用解析困难 |
| clangd LSP | `clangd_parser.py` | 与IDE行为一致，跨文件调用好 | 需clangd 20+，首次慢 |

### 1.3 图模型（实际实现 vs 设计文档）

**设计文档 (DESIGN.md) 声称采用"方案A"（极简）:**
- 仅 Function、Class 节点
- 不建 Repository、Directory、File 节点
- 目录信息由 `file_path` 推导

**实际实现 (graph_builder.py) 采用"方案B"（显式层级）:**
- Repository → Directory → File → Function/Class/Variable
- 完整的 CONTAINS 层级边

**这种不一致导致:**
- 维护困难（文档与代码不符）
- 增量更新逻辑更复杂（需维护更多节点类型）

## 二、当前方法的主要问题

### 2.1 ❌ 设计文档与实现不一致

```python
# graph_builder.py 实际构建的节点类型
nodes: dict[str, list[dict[str, Any]]] = {
    "Repository": [],    # 文档说不要
    "Directory": [],     # 文档说不要
    "File": [],          # 文档说不要
    "Function": functions,
    "Class": classes,
    "Variable": variables_list,
    "Attribute": [],
}
```

**影响:**
- 新开发者阅读文档后无法理解实际代码
- "方案A"的增量更新简单性优势未实现

### 2.2 ❌ 增量更新未真正实现

虽然记录了 `last_processed_commit`，但实际运行逻辑是：

```python
# run_stage1.py
print("清空现有代码图…")
clear_code_graph(driver)  # 全量删除
print("写入 Neo4j…")
write_graph(driver, graph)  # 全量写入
```

**缺失的能力:**
- `git diff` 获取变更文件列表
- 仅重解析变更文件的逻辑
- 图的局部更新（MERGE/DELETE 变更节点）

### 2.3 ❌ 跨文件调用关系解析不完善

**libclang 方案的问题:**
```python
# graph_builder.py
# 同一文件内按 name 解析被调用函数
# 跨文件仅当全局唯一 name 时再连（实际未实现）
search_path = callee_file_path if callee_file_path else file_path
if search_path in file_funcs_by_name and callee_name in file_funcs_by_name[search_path]:
    # 只能解析同文件内的调用
```

**clangd 方案的问题:**
- 需要 clangd 20+ 才支持 `callHierarchy/outgoingCalls`
- clangd 14~19 都会返回 CALLS=0

### 2.4 ❌ 存储层性能问题

```python
# neo4j_writer.py
# 逐条 MERGE，性能较差
for f in nodes.get("Function", []):
    session.run(
        "MERGE (n:Function {id: $id}) SET ...",
        ...
    )
```

**问题:**
- 没有使用 `UNWIND` 批量导入
- 大型仓库（如 llama.cpp）写入耗时较长

### 2.5 ❌ 缺少高层次抽象

| 缺失抽象 | 影响 |
|----------|------|
| Module 节点 | 无法直接回答"某个模块包含哪些子模块" |
| System/Subsystem | 无法表达架构层级 |
| Workflow 自动命名 | 阶段2生成Workflow但未在QA中深度使用 |

### 2.6 ❌ 变量引用分析有限

```python
# ast_parser.py 只收集简单变量引用
# 复杂的指针、引用、全局变量修改追踪未实现
```

## 三、改进建议

### 3.1 短期改进（1-2周）

#### A. 统一设计与实现

**选择1: 回到方案A（推荐）**

删除 Repository/Directory/File 节点，简化图模型：

```python
# 新 graph_builder.py
nodes = {
    "Function": [...],  # 带 file_path 属性
    "Class": [...],     # 带 file_path 属性
    "Variable": [...],  # 带 file_path 属性
}
# 删除 CONTAINS 层级边，只保留 CALLS、REFERENCES_VAR
```

**好处:**
- 与 DESIGN.md 一致
- 增量更新更简单（只需按 file_path 匹配）
- 查询依然可以通过 `file_path STARTS WITH` 实现目录过滤

**选择2: 接受方案B**

更新 DESIGN.md 承认实际采用方案B，并补充方案B的增量更新策略。

#### B. 优化 Neo4j 写入性能

使用批量导入：

```python
# 使用 UNWIND 批量写入

UNWIND $functions AS f
MERGE (n:Function {id: f.id})
SET n.name = f.name, n.file_path = f.file_path, ...
""", functions=func_list)
```

#### C. 改进跨文件调用解析

**方案1: 基于符号表的全局解析**

### 3.3 长期改进（2-3个月）

#### G. 支持多版本图

```python
# 支持图的版本管理
CREATE (f:Function {id: "xxx", version: "commit_abc123", ...})
# 或
CREATE (f:Function {id: "xxx"})-[:VERSION]->(v:Version {commit: "abc123"})
```

#### H. 语义层面的边类型

当前只有 CALLS，可以增加：
- `INHERITS_FROM`（类继承）
- `IMPLEMENTS`（实现接口）
- `OVERRIDES`（方法重写）
- `USES`（使用某个类/函数，非调用）

#### I. 基于 Embedding 的相似边

```python
# 为函数生成 embedding
# 创建相似函数之间的 SIMILAR_TO 边
MATCH (f1:Function), (f2:Function)
WHERE f1 <> f2 AND similarity(f1.embedding, f2.embedding) > 0.9
MERGE (f1)-[:SIMILAR_TO]->(f2)
```

## 五、具体代码改进示例

### 示例1: 批量写入 Neo4j

```python
# src/neo4j_writer.py

def write_graph_batch(driver, graph: dict[str, Any], batch_size: int = 1000) -> None:
    """批量写入，提升性能"""
    nodes = graph["nodes"]
    edges = graph["edges"]
    
    with driver.session(database=NEO4J_DATABASE) as session:
        # 批量写入 Function
        funcs = nodes.get("Function", [])
        for i in range(0, len(funcs), batch_size):
            batch = funcs[i:i+batch_size]
            session.run("""
                UNWIND $batch AS f
                MERGE (n:Function {id: f.id})
                SET n.name = f.name, n.signature = f.signature,
                    n.file_path = f.file_path, n.start_line = f.start_line,
                    n.end_line = f.end_line
            """, batch=batch)
        
        # 批量写入 CALLS 边
        calls = edges.get("CALLS", [])
        for i in range(0, len(calls), batch_size):
            batch = [{"from": c[0], "to": c[1]} for c in calls[i:i+batch_size]]
            session.run("""
                UNWIND $batch AS c
                MATCH (a:Function {id: c.from}), (b:Function {id: c.to})
                MERGE (a)-[:CALLS]->(b)
            """, batch=batch)
```

### 示例3: 增量更新框架

```python
# src/incremental_update.py

import subprocess
from typing import Set

def get_changed_files(repo_root: Path, since_commit: str) -> Set[str]:
    """获取自上次处理以来变更的文件"""
    result = subprocess.run(
        ["git", "diff", "--name-only", f"{since_commit}..HEAD"],
        cwd=repo_root,
        capture_output=True,
        text=True
    )
    return set(result.stdout.strip().split("\n"))

def update_graph_incremental(driver, build_dir: Path, repo_root: Path) -> None:
    """增量更新图"""
    last_commit = get_last_processed_commit(driver)
    current_commit = get_head_commit(repo_root)
    
    if last_commit == current_commit:
        print("无变更，跳过更新")
        return
    
    changed_files = get_changed_files(repo_root, last_commit)
    source_files = {f for f in changed_files if f.endswith(('.c', '.cpp', '.cc'))}
    
    print(f"变更文件: {len(source_files)} 个")
    
    for file_path in source_files:
        # 1. 删除旧节点
        delete_file_nodes(driver, file_path)
        
        # 2. 重新解析
        tu = parse_translation_unit(file_path, build_dir, repo_root)
        if tu:
            # 3. 写入新节点
            write_partial_graph(driver, tu)
    
    # 4. 更新 commit 记录
    update_last_processed_commit(driver, current_commit)
```

## 六、总结

当前建图方法的主要问题是**设计与实现不一致**以及**增量更新未完成**。建议：

1. **短期**（立即）：统一设计与实现，优化写入性能
2. **中期**（1个月）：实现增量更新，改进跨文件调用解析
3. **长期**（2-3个月）：添加 Module 抽象，丰富边类型

这些改进将显著提升：
- 建图速度（批量导入 + 增量更新）
- 图质量（跨文件调用 + 语义边）
- QA准确率（更好的检索能力）
