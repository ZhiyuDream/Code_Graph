## V7 Base - ReAct 基础版本

### 核心能力
1. **问题分析**: 使用 LLM 提取查询实体和关键概念
2. **混合检索**: 结合向量搜索（索引）和关键词搜索（Neo4j）
3. **ReAct 迭代**: 3-5 轮思考-行动-观察循环
4. **Answer 生成**: 基于检索到的函数和 Issue 回答问题

### 基础工具集
- `semantic_search`: 向量相似度搜索
- `graph_search`: Neo4j 关键词搜索
- `expand_callers/callees`: 调用链扩展（Neo4j）
- `issue_search`: GitHub Issue 搜索

## P0 - 优化版本 1 (最佳性能)

### 核心改进

#### 1. Fallback Grep 搜索
**问题**: Embedding 模型对某些关键词的检索效果差（相似度 < 0.5）

**解决方案**: 
```python
def should_trigger_fallback(question: str, functions: list) -> bool:
    """当最高相似度 < 0.5 或前10个平均 < 0.4 时触发"""
    if not functions:
        return True
    max_score = max(f.get('score', 0) for f in functions)
    top10_avg = sum(f.get('score', 0) for f in functions[:10]) / len(functions[:10])
    return max_score < 0.5 or (len(functions) >= 5 and top10_avg < 0.4)
```

**触发条件**:
- 初始检索函数 < 5 个，或
- 最高相似度 < 0.5，或  
- 前10个平均相似度 < 0.4

**实现**: 在 `scripts/run_qa_v7_optimized.py` 中 `smart_search()` 函数

#### 2. 智能停止机制
**问题**: 固定 3-5 轮迭代造成冗余计算，信息增益递减时仍继续

**解决方案**:
```python
# 连续2轮信息增益 <= 1 时熔断停止
if step >= 3 and len(info_gain_history) >= 2:
    recent_gains = info_gain_history[-2:]
    if all(g <= 1 for g in recent_gains):
        print(f"⚠️ 连续{len(recent_gains)}轮信息增益≤1，触发熔断停止")
        break
```

**逻辑**:
- Step 3+: 开始检测
- 连续 2 轮新增函数 <= 1 时停止
- 减少无效扩展，提升效率

## P1 - 智能路由版本

### 核心改进

#### 1. LLM 路由决策
**策略**: 使用 LLM 分析 Query 特征，选择检索策略

**策略选项**:
- `semantic`: 纯语义搜索
- `grep`: 纯关键词搜索
- `graph`: Neo4j 图查询
- `hybrid`: 混合策略（并行执行）

**路由逻辑**:
```python
def llm_route_decision(client, question: str) -> dict:
    """LLM 根据问题特征选择策略"""
    prompt = f"""分析以下问题，选择最佳检索策略：

问题：{question}

可选策略：
1. semantic - 概念理解、设计原理、功能描述
2. grep - 具体函数名、文件名、类名
3. graph - 依赖关系、调用链、模块层级
4. hybrid - 复杂问题，需要多角度

请输出 JSON：{{"strategy": "策略名", "entities": ["关键实体"], "reason": "选择原因"}}"""
```

#### 2. 多策略并行执行 (Hybrid)
```python
if strategy == "hybrid":
    # 并行执行多个策略
    with concurrent.futures.ThreadPoolExecutor() as executor:
        futures = []
        if "semantic" in secondary:
            futures.append(executor.submit(semantic_search, ...))
        if "grep" in secondary:
            futures.append(executor.submit(grep_search, ...))
        # 收集所有结果
```

## 关键洞察总结

### 1. 简单策略优于复杂路由
- P0 的简单混合策略（Semantic + Graph + Grep Fallback）优于 P1 的 LLM 路由
- 复杂路由增加了步骤和延迟，但准确率反而下降

### 2. 工具效率差异巨大
| 工具类型 | 效率评级 | 新函数/调用 |
|---------|---------|------------|
| Graph 搜索 | ⭐⭐⭐⭐⭐ | 6.12 |
| Grep Fallback | ⭐⭐⭐⭐⭐ | 5.97 |
| Semantic | ⭐⭐⭐⭐ | 5.73 |
| Issue 搜索 | ⭐⭐⭐ | 2.31 |
| Callers 扩展 | ⭐ | 0.55 |
| Callees 扩展 | ⭐ | 0.49 |

### 3. callers/callees 扩展的隐性价值
- 虽然 80% 调用失败，但剩余 20% 可能提供关键上下文
- 完全移除导致准确率下降 8.5%
- 应该优化触发条件而非完全移除

### 4. 智能停止是必需的
- 信息增益递减时及时停止，避免浪费
- P0 的熔断机制（连续2轮 <=1）有效减少冗余

---

## 下一步计划

1. **优化 callers/callees 触发条件**: 不移除扩展，而是智能选择何时扩展
2. **改进 P0 的 fallback 阈值**: 借鉴 P1 的优点，优化触发条件
3. **混合策略权重调优**: 平衡 semantic/graph/grep 的权重
4. **延迟优化**: 在保持准确率的前提下进一步降低延迟
