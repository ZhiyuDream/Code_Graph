# 变量图验证示例：用 Neo4j 回答“变量在哪里被用到”

本文档说明如何用图中的 **Variable** 节点和 **REFERENCES_VAR** 边验证“变量确实有用、能回答问题”。以 llama.cpp 中一段真实代码为例，按问题模板提问，并在 Neo4j 中执行 Cypher 核对。

---

## 0. 若 REFERENCES_VAR 查不到边时的诊断

若按 3.1 查「引用 cur 的函数」得到 **0 行**，可先做下面几条诊断（在 Neo4j 中执行）：

1. **REFERENCES_VAR 总数**  
   `MATCH ()-[r:REFERENCES_VAR]->() RETURN count(r);`

2. **边指向的 Variable id 长什么样**（抽几条看是否和 Variable 节点 id 一致）  
   `MATCH (f:Function)-[r:REFERENCES_VAR]->(v:Variable) RETURN v.id, f.name LIMIT 10;`

3. **该变量是否有入边**（把 id 换成你查到的 Variable id，如 `common/sampling.cpp:112:4:cur`）  
   `MATCH (v:Variable {id: 'common/sampling.cpp:112:4:cur'})<-[:REFERENCES_VAR]-() RETURN count(*);`

4. **sampling.cpp 里函数的 end_line**（若 end_line 等于 start_line，引用行可能落不进区间，归属会失败）  
   `MATCH (f:Function) WHERE f.file_path CONTAINS 'sampling.cpp' RETURN f.name, f.start_line, f.end_line ORDER BY f.start_line LIMIT 20;`

**常见原因**：clangd 的 documentSymbol 有时只给函数签名行，`end_line` 等于 `start_line`，导致“引用所在行”不在任何函数的 `[start_line, end_line]` 内。  
**处理**：在 `clangd_parser.py` 中已为引用归属加 fallback；且**声明行**（引用行 = 变量声明行）不再归属到任何函数，避免误归到错误函数。**重新跑一遍** `python run_stage1_clangd.py` 再查 3.1。

**结果解读**：若只看到 `ref_lines = [声明行]` 且被归到不相干的函数（如 `size`），说明 clangd 的 `textDocument/references` 对该符号只返回了声明、未返回使用点；此时图能回答“有谁引用过该变量”但行号不完整。若看到多条 `ref_lines` 且对应源码确为使用处，则图可用来回答“变量在哪些函数、哪些行被用到”。

## 1. 问题模板示例（变量类）

可抽象为：

- **Q1**：变量 `X` 在哪些函数里被引用？分别在哪些行？
- **Q2**：文件 `F` 中名为 `X` 的变量，在哪些位置被使用？
- **Q3**：函数 `func` 内部引用了哪些变量？各自在哪些行？

下面用 **Q1/Q2** 做一次完整验证。

---

## 2. 从 llama.cpp 选一个具体变量

以 **common/sampling.cpp** 为例（该文件在 compile_commands 中，已被阶段 1 建图）：

- **变量**：`cur` —— `common_sampler` 的成员 `std::vector<llama_token_data> cur`（约第 111 行）。
- **预期**：在同一个结构体的成员函数里被引用，例如 `set_logits` 中多处 `cur.resize(...)`、`cur[i] = ...`、`cur.data()` 等；可能还有其它函数引用它。

**验证思路**：在 Neo4j 中查出“名为 `cur`、且来自 sampling.cpp 的变量”被哪些 Function 通过 REFERENCES_VAR 引用，以及边上的 `lines`；再打开 `common/sampling.cpp` 对照这些行，看是否确实是 `cur` 的使用位置。

---

## 3. Cypher 查询（在 Neo4j Browser 中执行）

Neo4j 中 Variable 的 `id` 格式为：`file_path:start_line:start_character:name`。  
其中 `file_path` 为绝对路径（如 `/data/yulin/RUC/llama.cpp/common/sampling.cpp`）。  
REFERENCES_VAR 边属性：`lines`（引用发生的行号列表）。

### 3.1 按“变量名 + 文件”查：该变量在哪些函数、哪些行被引用

先确认该文件下名为 `cur` 的变量节点（可能有多条，如不同结构体/函数内的 `cur`）：

```cypher
MATCH (v:Variable)
WHERE v.name = 'cur' AND v.file_path CONTAINS 'sampling.cpp'
RETURN v.id, v.name, v.file_path, v.start_line
LIMIT 20
```

记下你要验证的那条 `v.id`（例如 sampling.cpp 里 common_sampler 的成员 `cur` 对应的 id）。

然后查“哪些函数引用了该变量，以及引用行号”：

```cypher
MATCH (f:Function)-[r:REFERENCES_VAR]->(v:Variable)
WHERE v.name = 'cur' AND v.file_path CONTAINS 'sampling.cpp'
RETURN f.name AS func_name, f.id AS func_id, f.file_path AS file_path, r.lines AS ref_lines
ORDER BY f.file_path, f.name
```

若图正确，应能看到：引用 `cur` 的函数名（如 `set_logits`、`reset` 等）、所在文件、以及 `ref_lines`（引用行号列表）。打开 `common/sampling.cpp` 对应行，应能看到对 `cur` 的读写。

### 3.2 按变量 id 精确查（已知某条 Variable 的 id 时）

若已知变量 id（例如从上面查询得到）：

```cypher
MATCH (f:Function)-[r:REFERENCES_VAR]->(v:Variable {id: $var_id})
RETURN f.name, f.file_path, r.lines
```

把 `$var_id` 替换为实际 id（或 Neo4j Browser 里用参数面板填 `var_id`）。

### 3.3 反向：某函数引用了哪些变量

例如查 `set_logits` 在 sampling.cpp 里引用了哪些变量：

```cypher
MATCH (f:Function)-[r:REFERENCES_VAR]->(v:Variable)
WHERE f.file_path CONTAINS 'sampling.cpp' AND f.name = 'set_logits'
RETURN v.name AS var_name, v.id AS var_id, r.lines AS ref_lines
```

---

## 4. 如何对照源码验证

1. 用 **3.1** 得到 `func_name`、`file_path`、`ref_lines`。
2. 打开 `file_path` 对应文件（如 `llama.cpp/common/sampling.cpp`），跳到 `ref_lines` 中的每一行。
3. 检查该行是否确实出现对变量 `cur` 的引用（如 `cur.resize`、`cur[i]`、`cur.data()` 等）。

若一致，说明图能帮助回答“变量在哪里被用到”；若某行明显不是对 `cur` 的引用，再检查是否为同名变量或建图/归属逻辑问题。

---

## 5. 小结

- **问题**：变量 X 在哪些函数、哪些行被引用？（模板中的变量类问题）
- **图的作用**：通过 Variable 节点 + REFERENCES_VAR（Function→Variable，属性 lines）直接查出“谁在哪些行引用了该变量”。
- **验证方式**：在 Neo4j 执行上述 Cypher，得到函数名与行号列表，再在源码中逐行核对。

以上均在 **Code_Graph** 阶段 1 使用 **run_stage1_clangd.py**（且 `collect_var_refs=True`）建图、并写入 Neo4j 的前提下有效。
