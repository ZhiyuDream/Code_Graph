# 实验: DeepSeek-v4-pro 模型测试

## 日期
2026-04-22

## 实验配置
- **模型**: DeepSeek-v4-pro
- **数据集**: results/qav2_test_cleaned.csv（324题，已排除36题空参考答案）
- **检索配置**: V8 + File Expansion + Grep V2
- **评估**: gpt-4.1-mini 作为统一 judge
- **并行**: 20 workers
- **API 修复**: 
  - `max_tokens` 默认 1000（DeepSeek 不支持 `max_completion_tokens`）
  - `timeout` 从 20s 提升到 60s（DeepSeek 推理较慢）

## 结果

### 准确率对比（相同324题，统一 gpt-4.1-mini judge）

| 模型 | 准确率 | 正确题数 |
|------|--------|---------|
| **GPT-5.4** | **87.0%** | 282/324 |
| **DeepSeek-v4-pro** | **53.1%** | 172/324 |
| **差距** | **-33.9%** | -110题 |

### 题目级对比

| 类别 | 数量 | 说明 |
|---|---|---|
| 两者都对 | 148 | 基础能力重叠 |
| 两者都错 | 18 | 共同难题 |
| **GPT-5.4 独有正确** | **134** | GPT-5.4 显著优势 |
| DeepSeek 独有正确 | 24 | DeepSeek 少量优势 |

### DeepSeek 错误原因统计

| 错误类型 | 数量 | 占比 |
|---------|------|------|
| **推断/推测过多** | **98** | **64.5%** |
| **未直接回答/偏离问题** | **52** | **34.2%** |
| 缺少关键信息 | 24 | 15.8% |
| 超出参考答案范围 | 7 | 4.6% |
| 事实错误 | 4 | 2.6% |

## 典型对比案例

### DeepSeek 失败案例

**[1] apertus 子模块层级关系**
- GPT-5.4 ✅: "详细分析了 apertus 相关的具体组成部分及层级关系"
- DeepSeek ❌: "转向了与 apertus 无关的 autoparser 组件分析"（**概念混淆**）

**[6] falcon 依赖关系**
- GPT-5.4 ✅: "准确分析了 falcon 实现依赖的模块"
- DeepSeek ❌: "参考答案指出无法确认 falcon 有显式依赖，而生成答案却列举了大量具体依赖"（**推断过度**）

**[8] commonpp 依赖关系**
- GPT-5.4 ✅: "从整体架构层面分析了 commonpp 的依赖"
- DeepSeek ❌: "讨论的是 common.cpp 模块及其依赖，与问题询问的 commonpp 无关"（**概念混淆**）

### DeepSeek 优势案例

**[51] ggml-alloc.c 算法实现**
- DeepSeek ✅: "详细分析了动态张量内存分配算法的核心函数及处理流程"
- GPT-5.4 ❌: "未直接回应问题核心内容"

**[59] VDR_Q2_K_Q8_1_MMQ 依赖关系**
- DeepSeek ✅: "合理推断了流水线依赖关系、执行优先级及协同工作方式"
- GPT-5.4 ❌: "内容完全偏离了题目"

## 关键发现

### 1. DeepSeek 推断能力弱于 GPT-5.4

**核心差异**: 两者都喜欢推断，但 GPT-5.4 的推断更精准，DeepSeek 的推断经常"跑偏"。

| 维度 | GPT-5.4 | DeepSeek |
|---|---|---|
| 推断准确性 | 高，贴合参考答案 | 低，经常偏离 |
| 概念辨识 | 能区分相似概念 | 容易混淆（apertus↔autoparser） |
| 问题聚焦 | 始终围绕问题核心 | 经常转向无关内容 |

### 2. DeepSeek 的 reasoning 机制是双刃剑

DeepSeek-v4-pro 有显式的 reasoning_content（思考过程），这导致：
- **优点**: 思考过程更透明
- **缺点**: 
  - 占用大量 token（600+），导致 content 容易被截断
  - 推理过程暴露后，judge 更容易发现推断漏洞

### 3. API 兼容性成本

| 问题 | 影响 | 解决方案 |
|---|---|---|
| 推理过程占 token | content 被截断 | max_tokens 提升至 1500+ |
| 响应慢（40-50s/题） | 实验耗时增加 3x | timeout 提升至 60s |
| 不支持 max_completion_tokens | 需用 max_tokens | 条件判断 |

## 结论

**DeepSeek-v4-pro 在该 benchmark 上表现显著弱于 GPT-5.4**

- 准确率差距: -33.9%（53.1% vs 87.0%）
- 核心差距: 推断准确性、概念辨识能力、问题聚焦能力
- DeepSeek 仅在少数具体实现细节问题上略有优势

## 建议

### 短期
- **继续采用 GPT-5.4** 作为主力生成模型
- DeepSeek 可作为辅助（如需要显式 reasoning 的场景）

### 中期
- 若需使用 DeepSeek，需优化 prompt：
  - 明确要求"只基于检索到的信息回答"
  - 禁止推断超出检索范围的内容
  - 要求先确认问题中的核心概念再回答

### 长期
- 探索 DeepSeek 的推理优势（如数学、逻辑题）是否与代码理解任务匹配
- 考虑使用 DeepSeek 的 reasoning 能力辅助检索策略优化
